# BUGate Full Check

## Purpose

Use this skill to verify a BUGate checkout after setup or before mounting a real
SUT. It distinguishes:

- installed binaries and services,
- BUGate core/demo capability,
- optional runtime capability,
- profile-activated gates for a real SUT.

Do not claim "all gates are activated for a real SUT" unless an active SUT
profile is mounted and guarded paths/resources are configured. In core mode,
report that core/demo/optional runtime checks are available, not that a real SUT
is protected.

## Required Context

1. Read `.shared/skills/bugate/SKILL.md` first.
2. Inspect `bugate.config.yaml` or the active `BUGATE_PROFILE`.
3. Keep all generated probe artifacts in `/tmp` or ignored runtime dirs.
4. Do not invent product facts, SUT API paths, secrets, or environment names.

## Quick Start

For a low-cost health check that does not invoke real peer model dispatch:

```bash
python3 .shared/skills/bugate-full-check/scripts/run_full_check.py --mode smoke
```

For the full post-install audit, including real Codex and Claude peer dispatch:

```bash
python3 .shared/skills/bugate-full-check/scripts/run_full_check.py --mode full
```

Use `--timeout-seconds <n>` if peer dispatch needs a longer per-command window.

## Interpretation Rules

- `check-env` only proves `codex` and `claude` binaries resolve. It does not
  prove authentication. The full check also runs `codex exec` and `claude -p`.
- A Stage 3B adversarial file generated by the bridge may remain
  `gate_status: pending`; that is expected until human review accepts it.
- Wave 7 role isolation is default-off until `BUGATE_AGENT_ROLE` and profile
  `agent_roles` are set.
- Physical write guarding is default-off in core mode because
  `guarded_path_regex: []`; mounted profiles turn it on.
- Wave 9 change-driven regeneration is currently methodology-level unless the
  mounted SUT supplies stale-detection tooling.

## Expected Final Answer

After running the check, summarize results in four groups:

- Installed and verified usable.
- Available but requiring a real SUT profile to activate.
- Gates that intentionally require human acceptance.
- Methodology-only or SUT-profile-owned parts.

Finish with a precise conclusion such as:

> BUGate core plus optional runtimes are verified. This does not yet mean a real
> SUT's gates are activated, because the active config is still core mode.

or:

> BUGate is mounted to `<profile>` and the configured guarded paths/profile gates
> are active.
