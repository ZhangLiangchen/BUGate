# BUGate Full Check

## Purpose

Use this skill to verify a BUGate checkout after setup or before importing it
into a real SUT test repo. It distinguishes:

- installed binaries and services,
- BUGate core/demo capability,
- runtime capability — required memory bus, optional dual-agent CLIs and
  profile-defined path isolation, plus the opt-in auditable Wave 7 lifecycle,
- profile-activated gates for a real SUT.

Do not claim "all gates are activated for a real SUT" unless an imported SUT
profile is active and guarded paths/resources are configured. In core mode,
report that core/demo/runtime checks are available, not that a real SUT
is protected.

The outer layout also changes the installation check:

- In **imported mode**, full-check first runs the selected vendored engine's
  `bin/bugate-update verify` against the workspace and its actual relative
  vendor directory. The run fails unless the release manifest and committed
  installed lock verify, the installation is lock-based, and no interrupted
  transaction requires recovery. A failed preflight stops immediately before
  compiling or executing any other vendored probe. Custom in-repo vendor
  directories are supported.
- In **core/release mode**, there is no imported installed lock to verify.
  Full-check validates the available Core capability directly; this does not
  turn the checkout into an imported installation.

## Required Context

1. Read `.shared/skills/bugate/SKILL.md` first.
2. Inspect `bugate.config.yaml` or the active `BUGATE_PROFILE`.
3. Keep all generated probe artifacts in `/tmp` or ignored runtime dirs.
4. Do not invent product facts, SUT API paths, secrets, or environment names.

## Quick Start

For a low-cost health check that does not invoke real peer model dispatch:

```bash
python3 .shared/skills/bugate-full-check/scripts/run_full_check.py --mode smoke
```

Smoke mode still runs the complete required-mode Wave 7 chain against a
temporary SUT-neutral imported workspace and the real Memory service under the
dedicated `project:bugate-full-check` namespace: unset and
wrong-role negatives, synthetic human-acceptance record, strict handoffs,
exact-ID acceptances in distinct sessions, Layer 4 hook controls, guarded
implementation handoff, reviewer acceptance, post-run, completion, and strict
final verification. It skips only real model peer dispatch.

For the full post-install audit, including real Codex and Claude peer dispatch:

```bash
python3 .shared/skills/bugate-full-check/scripts/run_full_check.py --mode full
```

Use `--timeout-seconds <n>` if peer dispatch needs a longer per-command window.
The real-peer fixture runs from a temporary SUT-neutral CWD with its own
mode-off profile and Memory namespace. For that non-git CWD only, full-check
sets the bridges' explicit `SDTD_CODEX_SKIP_GIT_REPO_CHECK=1` automation opt-in;
ordinary imported dispatch keeps Codex's repository check.

## Interpretation Rules

- An imported full-check cannot pass on a legacy, missing, locally modified,
  manifest-mismatched, or recovery-required installed state. Updater failure,
  non-zero exit, malformed JSON, or any result other than `decision: GO`,
  `status: passed`, `installed_kind: locked`, `lock_based: true`, and
  `recovery_required: false` is a hard failure, never a warning.
- `check-env` only proves `codex` and `claude` binaries resolve. It does not
  prove authentication. The full check also runs `codex exec` and `claude -p`.
- A Stage 3B adversarial file generated by the bridge may remain
  `gate_status: pending`; that is expected until human review accepts it.
- Wave 7 has two independent controls. `agent_roles` path isolation remains a
  default-off env/profile policy. Auditable lifecycle governance remains inert
  when `role_governance.mode: off`, warns in `advisory`, and fails closed in
  `required` for missing/wrong role/session, missing/stale receipts, or drift.
- Wave 1 real Codex/Claude peer dispatch is exercised only in full mode. Those
  peers are not Wave 7 lifecycle actors and must not inherit parent role/session
  identity.
- Synthetic template, guard, hardening, dialect, role-flow, and real-peer
  fixtures must not inherit an imported SUT profile, project CWD, or Memory
  namespace. The active imported profile is consulted only by checks that
  explicitly report its activated Wave 0/8 and activation-boundary state.
- The role-flow probe resolves every command from the selected engine root, so
  a SUT-owned `scripts/` tree cannot shadow a vendored engine. Treat imported
  and core root-resolution regressions as failures.
- Ordinary Memory recall/heartbeat remains best-effort, but the Wave 7 probe's
  strict Memory transitions must fail closed and exact-verify their content
  hashes. A down Memory service is therefore a full-check failure, not a reason
  to downgrade the gate.
- Physical write guarding is default-off in core mode because
  `guarded_path_regex: []`; an imported SUT profile turns it on.
- Wave 9 change-driven regeneration is currently methodology-level unless the
  imported SUT repo supplies stale-detection tooling.

## Expected Final Answer

After running the check, summarize results in four groups:

- Installed and verified usable.
- Available but requiring a real SUT profile to activate.
- Gates that intentionally require human acceptance.
- Methodology-only or SUT-profile-owned parts.

Also report whether the required-mode temporary role chain reached `closed`,
whether strict Memory exact-ID checks passed, and whether the active imported
profile actually enables `role_governance`. For imported mode, report that the
lock/manifest/updater/recovery preflight passed. For core mode, report that this
preflight is intentionally not applicable. A synthetic full-check flow proves
the engine contract; it does not prove that a real UC has been accepted.

Finish with a precise conclusion such as:

> BUGate core, the required memory bus, and the optional runtimes are verified.
> This does not yet mean a real SUT's gates are activated, because the active
> config is still core mode.

or:

> BUGate is imported into the `<profile>` SUT repo and the configured guarded
> paths/profile gates are active.
