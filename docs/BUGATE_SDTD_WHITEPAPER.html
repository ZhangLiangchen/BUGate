第 1 章

远景与目标:把"AI 写测试"从手艺变成工程

BUGate 想解决的不是"让 AI 写出更多测试",而是"让 AI 写出能发现缺陷的测试,并且这件事可被任何 QA 可靠复现,而不依赖资深个人英雄"。

1.1 为什么需要它:AI 测试开发的四类系统性失败

让大语言模型直接从需求文档生成端到端测试,会系统性地产生四类失败。它们成因不同、防御策略不同,任何一类不治都会让"高覆盖率、高通过率"的测试对真实缺陷的发现能力趋近于零。

失败模式典型表现BUGate 的对策
浅业务模型对需求的理解停留在字面,漏掉状态机、边界、副作用风险;单一模型还会"一致性地"漏掉同一类东西Wave 1 双模型独立互审 + 分歧报告;Layer 1 命题化;1A/1B 域模型与状态流显式建模
假绿断言写宽(!=200、双类型 raises)、适配后端错误行为、HTTP 200 当业务成功宪法第 5 条;Layer 3 断言语义门;Wave 8 用真实证据改错来证明断言"能失败"
臆造数据编造资源 ID、地址、密钥、账号等,污染共享环境Layer 2 固定资源决策;Layer 3 数据来源必须挂 probe 证据;测试纪律硬规则
知识蒸发踩过的坑、确认的契约、发现的缺陷只留在聊天记录,下个会话从零开始Memory Bus 主记忆层;04 执行报告 / 05 知识回写;经验晋升闭环

上表是整套方法论的设计起点。它给出一个关键判断:这四类失败靠"提醒模型注意"是治不住的——因为 LLM 的训练目标里,"自信猜测"被奖励得多于"承认不知","能跑通"被奖励得多于"能发现缺陷"。唯一可靠的办法,是把约束做成机器可校验的产物加物理门禁。这正是 BUGate 名字的由来:Business Understanding Gate,先过业务理解的门,再谈代码。

1.2 远景:受治理的、可审计的自动化

必须诚实地界定远景,否则会误导使用者。BUGate 的目标不是"全自动产出可直接合入的测试代码"——那恰好会踩中上面四类失败中的"假绿"和"臆造数据"。它的真实定位是:

一句话定位:受治理的(human-in-the-loop)、双 runtime 互审的、黑盒的、规格驱动的测试用例开发流水线。AI 干脏活(起草、生成、跑门禁),人保留所有否决权;每一次"晋升"都由人或被人明确授权的 agent 做出。

这个定位里的每个限定词都是有意的:

  • 受治理:全流程保留 6 个人工审核点(见第 11 章),编排器被明文禁止自动晋升任何审核类产物、自动写测试代码、绕过门禁。
  • 双 runtime 互审:在需求理解(Wave 1)与对抗(Stage 3B)两个最易偏读的阶段,由 Codex 与 Claude Code 两个不同厂商的模型互不沾染地独立产出,分歧本身即风险信号。
  • 黑盒:API-first,不依赖被测系统源码白盒插桩;质量证据用 oracle falsification 而非白盒变异(理由见第 9 章)。
  • 规格驱动:任何测试代码诞生前,需求必须先被转换成可被机器校验、可被人审阅的规格产物。

1.3 它适合谁、不适合谁

BUGate 适合:有相对可靠需求文档(PRD/提测单)、以 API 黑盒 E2E 为主、且引入了 AI Agent(Codex/Claude Code 一类)参与测试开发的团队。它不是银弹——无法解决组织文化问题、完全未文档化的隐性知识、棕地系统存量测试迁移,这些在配套方法论里被明确划为范围之外。把预期对齐到"它让 AI 辅助测试从依赖资深经验的不稳定实践,变成任何 QA 都能可靠执行的标准化流程",是正确的使用姿态。

第 2 章

方法论支撑:站在 40 年测试研究的肩膀上

BUGate 的工程实现背后,是一套 SUT 无关的零领域 QA 方法论 METHOD(9 个 Wave、双层架构)。这套方法论不是灵感一闪——它的 7/9 个 Wave 直接坐落在有标准、有奠基论文的经典测试方法论之上;原创之处在于"面向 LLM 失败模式的重新编排",而非重新发明各项技术。

2.1 双层九 Wave 结构

METHOD 把测试开发拆成两层防御,对应 1.1 节的两类成因不同的失败:

层Wave解决什么
业务理解审计层Wave 0 PRD 健康度体检;Wave 1 多 AI 独立提取命题;Wave 2 引用回溯审计;Wave 3 结构化访谈;Wave 4 行为 oracle 验证防"浅业务模型":在写任何测试前,把需求转成有据、可追溯、分歧已处置的业务命题
缺陷发现生成层Wave 5 边界/状态机/风险加权;Wave 6 对抗式红队;Wave 7 三层 agent 隔离;Wave 8 质量门(变异/oracle falsification)防"缺陷发现弱":强制测试跳出 happy path,并用客观指标证明断言确实能失败

两层必须同时存在——即使业务理解完全正确,LLM 生成的测试仍倾向覆盖正向场景;反之,即使技术上很会找 bug,基于错误业务模型找到的"缺陷"也是噪音。这个"双层缺一不可"是 METHOD 最核心的结构性主张。

2.2 经典方法论谱系(理论基础与相关工作)

下表把 9 个 Wave 逐条映射到它所基于的经典测试方法论与奠基文献/标准。这是开源诚信归因的核心——它让读者知道每个 Wave 背后有数十年积累,也避免"用了变异测试/oracle 的名字却不引用来源"的尴尬。

Wave所基于的经典方法论奠基/标准出处
0 PRD 健康度需求可测性 / 需求静态测试ISO/IEC/IEEE 29148;ISTQB 静态测试
1 多 AI 独立提取独立测试设计与评审多样性;规格导出测试IEEE 1028;spec-based testing
2 引用回溯审计需求可追溯性矩阵ISO/IEC/IEEE 29148;DO-178C 双向可追溯
3 结构化访谈需求获取(elicitation)结构化访谈需求工程经典技术
4 行为 oracle测试预言问题Weyuker 1982《On Testing Non-testable Programs》;Barr 等 2015 oracle survey
5 边界/状态机/风险边界值分析、状态迁移测试、风险驱动测试Myers 1979《The Art of Software Testing》;Beizer 1990《Software Testing Techniques》;ISO 29119
6 对抗式红队负向测试、模糊测试、基于性质的测试Miller 1990(fuzzing);Claessen & Hughes 2000(QuickCheck)
7 三层 agent 隔离独立验证与确认(IV&V)IEEE 1012
8 质量门变异测试DeMillo, Lipton & Sayward 1978

这张表的读法:7/9 个 Wave(0、2、4、5、6、7、8)有明确的标准或奠基论文支撑;Wave 1、Wave 3 是把经典评审与需求获取实践编排进 AI 流水线的新组合。换言之,地基是公认的,楼是原创的。需要正式发表时,年份/卷期应按目标期刊格式核对补全页码与 DOI——本文只列可确证的奠基文献,不含臆造引用。

2.3 真正的原创贡献

经典方法论解决的是"如何设计有效测试";METHOD 新增解决的是"当生成测试的是 LLM 时,如何防止它在业务理解和缺陷发现两个环节系统性失败"。这部分在既有测试文献中没有现成答案,是真正的增量:

  • 以 AI 两类失败模式驱动设计——双层架构不是测试领域现成的,是按"LLM 会怎么失败"重新编排经典技术的结果。
  • "验证有据性 ≠ 验证正确性"的认知反转——QA 的工作从"判断 AI 对错"变为"审计 AI 引用是否有据",这是针对 AI 语境的认识论重构。
  • "多 AI 分歧 = 不确定性所在"的分歧路由——把评审多样性用到 LLM 上,分歧成为风险信号而非噪音。
  • "零领域知识 QA"定位——四个新角色(审计引用、路由分歧、采访开发、机械比对)都不要求领域知识,只要求流程纪律。
给开源的归因口径:对外应表述为 "original orchestration of established test methodology"(对既有测试方法论的原创编排与综合),不要用"全新发明"。这既诚实,又比"民间重新发明"可信得多。

第 5 章

整体架构:四层结构

BUGate 是一套四层结构,自上而下分别回答"为什么、做什么、怎么强制、怎么沉淀"。这个分层是理解整套系统的骨架。

图 5-1 BUGate 四层结构:方法论 → 工程流程 → 强制执行 → 记忆知识

逐层解释如下。

① 方法论层是"为什么"。它是 SUT 无关的 METHOD 9-Wave,提供整套流程的理论依据。工程层每一个具体门禁,都能追溯到这一层的某个 Wave(映射见 docs/qa-methodology/METHOD.md)。当某条工程规则看起来武断时,读方法论层就能找到"为什么"。

② 工程流程层是"做什么"。它把 9 个 Wave 压成 BUGate 四层(业务理解→可测性→用例蓝图→代码),并在 Full SDTD 扩展出10 阶段流水线。关键设计是产物即接口:阶段之间只通过带 frontmatter 的文件交接,一个 UC 一个目录、与测试文件 1:1,没有"AI 记得上一步说了什么"这种隐式状态。这让任何阶段可以换人、换模型、换会话重做。

③ 强制执行层是"怎么强制",也是 BUGate 区别于纯文档方法论的关键。它把约束分成两种:语义门(脚本 exit 0/非 0)管"产物内容是否合格",可被人 review 后接受;物理门(PreToolUse hook)管"没过门就连写代码的动作都执行不了",不接受任何口头理由。绕过物理门本身被定义为违规而非变通。

④ 记忆与知识层是"怎么沉淀"。Memory Bus 是跨 agent 的长期记忆 SSOT,带治理门(无证据的 confirmed 结论会被拒写);执行结果通过 04/05 回写,缺陷进 incidents 库,反复踩的坑晋升为 harness 资产。这一层让"知识蒸发"这类失败被结构性地堵住。

5.5 解耦模型:Core / Profile / Mounted Workspace / SUT Runtime

上面四层是 BUGate 引擎内部的纵向分层(为什么→做什么→怎么强制→怎么沉淀)。它与另一个正交的问题——BUGate 与被测系统如何解耦——是两回事:二者都叫"层/部分"但含义不同,务必区分。独立拆分后,BUGate 与一个 SUT 的关系是四部分:

部分归属内容
BUGate Core本仓库(SUT 无关)方法论、模板、四层语义门、物理写入门、双端 adapter、Wave 0 / Wave 8 评分器
SUT Profile桥接契约artifact_dir、guarded_path_regex、命令、证据来源、agent_roles、falsification/prd 规格、namespace
Mounted Workspace通常即 SUT 的自动化测试框架测试代码、BUGate 产物、fixtures、runners、生成用例、捕获证据、本地测试规则
SUT / Product Runtime产品方所有黑盒 API/UI/运行时行为、产品文档/契约/环境;可选 source/API dump 仅作证据源

关键纪律:"挂载一个 SUT"挂的是它的自动化测试框架 / 测试工作区,不是把产品源码、密钥、环境并进核心;Runtime 永远只作为黑盒证据被观测。挂载的具体机制见第 12 章。

6.2 记忆总线细分架构

记忆层是重要的外部服务,值得单独展开。它分三层:接入层(双端 hooks + memory_bus.py 包装 + progress.py)经带鉴权的本地 HTTP 调用服务进程;服务进程(mcp-memory-service,uvicorn/FastAPI)内含 REST API、治理门、以及一个三级嵌入路径(ONNX 主路径 → torch 备路径 → hash 兜底);底层用 sqlite-vec 存储 384 维归一化向量。

三级嵌入路径的设计:① ONNX 主路径(onnxruntime,已激活,PyTorch-free,footprint 由 torch 时代 2.5GB 降至数百 MB);② torch 备路径(sentence-transformers,失败自动回退);③ 纯 Python hash 兜底(前两者皆缺时,质量降级)。ONNX 与 torch 数值等价(同文本余弦 1.0),保证新旧向量同一空间。Memory Bus 是可选记忆运行时(非门禁必需),详见 docs/SETUP-OPTIONAL.md。

记忆总线的治理门是这套系统"知识可信"的关键:confirmed 状态的 finding 若不带证据(refs/evidence/artifact)会被直接拒写。这把 Memory Bus 从"杂乱文本池"约束成"可信结论库",让下游 agent 在会话启动时拿到的恢复上下文是经过质量门的。记忆总线的接入 / 服务 / 嵌入 / 存储分层详见 docs/SETUP-OPTIONAL.md。

第 7 章

各环节实现机制

本章逐阶段说明实现机制。每个阶段按"回答什么问题 → 产物 → 门禁怎么验 → 预期效果"展开。

7.1 Wave 1 — 双模型多视角需求理解

回答什么:单一 AI 读需求有一致性盲区,它漏掉的边界自己复查一百遍也还是漏。两个不同厂商的模型在互不沾染对方结论的前提下独立读同一份输入,分歧本身就是风险信号。产物:00_multiview/ 下的 input pack(只含原始材料引用,不含当前 agent 分析)、双方独立 view、divergence report、带模型/推理档/耗时/prompt sha256 的全程 metadata。门禁:sdtd_multiview.py check 校验两份独立 view、正式分歧报告、Layer 1 吸收决议齐备;对 Full SDTD UC,这个检查被 Layer 1 语义门强制内嵌——Wave 1 没走完,Layer 1 标不了 passed。预期效果:把"需求理解偏差"从不可见变成有产物、有归档、必须处置的显式分歧。CLI bridge 在调对端 CLI 时请求最强推理档、捕获输出后做 schema 校验,校验不过不落盘、失败件归档——这保证了互审不是走过场。

7.2 Layer 1 — 业务理解门 + 1A 域模型 / 1B 状态流

回答什么:这个需求在业务上到底说了什么?SUT 边界在哪?哪些是事实、哪些是推断、哪些根本没人知道?产物:01_business_brief.md,核心是 PRD Health 体检(8 维评分 + 可追溯性加成)、P-xxx 命题表(每条带原文逐字引用或行号/symbol/json-path 锚点)、O-xxx 业务 oracle 表、未知项清单(高危未答阻塞通过)、三档证据标注;以及 Full SDTD 的 01a_domain_model(业务对象/属性/关系/不变量)、01b_state_flow(状态/合法迁移/终态/异步/禁止迁移)。门禁:check_bugate_brief_semantics.py(含分档可验证性比率,≥0.80 健康 / <0.40 FAIL)。预期效果:下游全部检查都建立在命题之上——Layer 1 写歪,后面每层都会"忠实地实现错误的模型",所以这里是全链路最重的人工审核点。

7.3 Layer 2 + 2A — 可测性与测试层决策

回答什么:每条命题用什么层级测最便宜(API/契约/集成/E2E)?动不动真实资源?产物:02_testability.md(固定资源与 query-first 策略、marker 与环境、skip/xfail 边界、probe 计划)+ 02a_test_dimension_matrix.yaml(功能/状态/边界/权限/错误/异步/副作用风险七类维度显式拆解)。门禁:check_bugate_layer2_semantics.py(含命题覆盖:每个 P-xxx 进决策表,与 oracle-验收映射)。预期效果:不是每条命题都值得 E2E;E2E 用例必须说清为什么更便宜的层不够。资源纪律在这里锁定——日常回归 query-first 复用固定资源,只有创建类用例才允许建资源且必须打 marker。

7.4 Layer 3 — 用例蓝图

回答什么:具体测哪些 case?每条 case 绑哪些命题和 oracle?数据从哪来?断言是哪个函数?产物:03_inventory.yaml。每条 case 必须有 case ID、proposition_refs(必填)、oracle_refs(P0/P1 必填)、优先级、API 序列、data_source(probe 文件路径 + 五枚举 probe_status)、断言函数名(必须真实存在)、副作用分类、marker、skip/xfail 条件、incident 引用。门禁:check_bugate_inventory_semantics.py,机器校验 endpoint/client 路径真实存在、断言非 stub、probe 状态、marker 注册、反向命题覆盖、意图回溯。预期效果:杜绝"逻辑端点 + 占位断言 + 假装有数据"三件套。

7.5 Stage 3A / 3B — 人读用例与对抗用例(均为 Layer 4 物理前置)

3A 中文可读用例:从 03_inventory 生成中文优先的 03a_test_cases.md(全链路唯一中文优先产物),默认 pending,必须人工确认意图/前置/预期/副作用/skip-xfail 边界后才 passed。它给不读 YAML 的业务/测试同学一个低成本否决点——代码还没写,改主意最便宜的时刻。3B 对抗用例:复用 Wave 1 的双端机制,Codex 与 Claude 各自基于 01-03A 产物独立产出对抗视角,再合成 03b_adversarial_cases.yaml;每条带威胁模型、攻击路径、双端候选溯源、人工裁决。它补"命题导向天然偏 happy path"的盲区。这两个产物对 Full SDTD UC 都是物理写入门的前置条件。

7.6 Layer 4 — 物理写入门 + 代码实现

对 test_uc_*.py 的任何写入(双端),PreToolUse hook 先调 check_bugate.py:① 从文件名推 UC 目录,同号多目录时按 slug 精确绑定、无匹配 fail-closed;② 校验 L1/L2/L3 全部 passed;③ 对 Full SDTD UC(brief 或 inventory 声明该产物集),03a 与 03b 也必须 passed;④ 任一不满足即阻断写入。负向对照(无 UC 目录 / 前序未 passed)实测被阻断;examples/mounted-demo/ 走查可见 block→allow 行为。代码实现纪律要求复用既有 client/fixture/waiter/断言,完成判定必须落在业务终态 + 可观测证据上,HTTP 200 不算业务成功。

7.7 执行纪律 + 失败自愈 + 04/05 知识闭环

执行纪律:PASS/FAIL/XFAIL/SKIP 四种信号谁也不许冒充谁;已知只读缺陷用 xfail(修复后变 unexpected pass 自动报警),写副作用缺陷用 skip + bug 编号 + 恢复条件;禁止为绿适配后端错误行为。自愈:self_healing_mvp.py 失败后做分类(基建不稳/测试侧/SUT 缺陷三族,内置"排除 fixture/数据/鉴权/环境之前不得立 SUT 缺陷"的 human gate)、重跑决策、修复边界、补丁计划——但明确不自动改测试。04/05:generate_sdtd_reports.py 生成执行报告与知识回写草稿,人工校正后晋升;04 留执行证据,05 留持久结论并实时写 Memory。这一层让"知识蒸发"被结构性堵住。

诚实标注(起源期 · Hypervise,详见第 10 章):orchestrator --auto 已在真实 UC 上端到端驱动过——UC-AE-24(2026-06-12)的 pre-code DAG(Wave1 检查→L1/L2/L3 语义门→3A 生成,在人工审核点停下)与 post-run DAG(self-healing→失败行动计划→04/05→v13 门,在晋升点停下)均由 --auto 执行,UC-AE-24 通过全量审计。self-healing 也已在真实 staging pytest 日志上运行并产出失败行动计划。剩余的唯一缺口是:UC-AE-24 那一轮 exit 0 无失败,所以"真实失败 → 分类 → 重跑 → 修复"这条分支尚未在一次真正失败的运行上触发。见第 11 章。
第 8 章

五条宪法:不可违反的第一性原则

在四层语义门、物理写入门与若干 hook 之上,有 5 条不可违反的第一性原则。当任何单项检查、模板或脚本看起来允许违反时,宪法赢。它的存在是为了防止门禁被一项一项地"优化"掉,直到失去原本的作用。

条款内容
不变量 1
业务含义先于实现		先确立业务上的"对/错",再谈实现细节;用户故事、验收标准、PRD 原文、开发口头答复都是输入,只有转换成带 source_quote 锚点的 P-xxx 命题、带可观测面的 O-xxx oracle 之后才能变成工件。
不变量 2
命题可测 / 有界 / 显式延后		每条重要命题要么可测、要么有明确边界、要么被显式标注延后;不允许悬空的模糊主张混进规格。
不变量 3
用例可回溯命题与 oracle		每条 Layer 3 case 回溯到 Layer 1 的 P-xxx 命题与(适用时)业务 O-xxx oracle;断言 → case → 命题 → 原文引用/probe 字段,任何一环断了就不允许进代码。HTTP 200 永远不是业务 oracle。
不变量 4
证据分档不升级		每个业务数据标 fact / inferred / unknown 三档;为让门禁通过而把 inferred/unknown 改标成 fact 是被禁止的。缺证据的正确动作是 probe、提问或移出范围。
不变量 5
绕过预期的绿不算质量		一次绿色运行如果绕过了被测的预期行为,就不是质量证据。宽松断言、适配错误行为、HTTP-only 成功、假数据都违反本条。xfail 意外变 pass、skip 被移除都是真实工作发生的信号,必须保留。

这五条的精神可以浓缩为一句话:诚实优先于绿色。前两条保证"测的是业务真正关心的事、且可测可追溯";第三、四条保证"证据有据可查、不被升级粉饰";第五条保证"测试存在的目的是发现缺陷而非制造通过"——正好对应第 2 章的双层结构。"理解没到位连代码都写不了"不是这五条不变量本身,而是由物理写入门 hook 强制落地的工程纪律(见第 6、7 章)。这五条不变量逐字收录在 references/sdtd-constitution.md,Codex 与 Claude 的 gate agent 在评审任何一层前都先加载它。

第 9 章

质量证据:断言有效性第一次有了分数

宪法第 5 条 说"测试为发现 bug",但这本身只是声明。Wave 8 的任务是给它一个机器化、客观、可证伪的指标——回答"现有断言在真实业务证据被故意改错时,能否稳定失败"。

9.1 路线决策:为什么不做白盒变异

Wave 8 的经典形态是白盒 SUT 变异测试(改后端源码植入缺陷,看测试能否杀死)。BUGate 明确不走这条路:面向黑盒 API-first E2E,改后端实现、重新部署、隔离环境、清理副作用、归因环境波动,成本高且归因不稳。替代方案是更适配黑盒的 oracle falsification / evidence replay:取真实捕获的业务证据快照,按字段级改错(状态倒退、金额错配、地址错配、删关键字段等),离线调用断言,验证"错误的业务状态必须被杀死"。这是一个有意的、有决策文档背书的路线选择,而非能力缺失。

9.2 首份真实结果

在中立的 examples/demo-sut 上,同一引擎自检得分 85.7%(killed 6 / survived 1),证明评分器 SUT 无关地可用。下面是起源期在真实业务证据上跑出的首份结果(Hypervise,详见第 10 章)。

图 9-1 Wave 8 oracle falsification 起源期首份真实结果(Hypervise · UC-LA-11,2026-06-11)

判定语义是:某个改错被任一绑定 oracle 杀死即算 killed,全部 oracle 都放过才算 survived;每份证据先过基线(未改错时全部断言必须通过),否则该证据记为 not_run。这保证分数有意义——只针对断言本该接受的正确证据来检验它对错误证据的杀伤力。14 个 survived 不是失败,而是真实缺口的诚实暴露(起源期暴露了两类共享断言缺口,详见第 10 章),每条都带显式处置选项:补共享断言、补用例、或显式接受非严格边界。

9.3 断言覆盖矩阵

Wave 8 的另一半是断言覆盖矩阵,反查 R-xx 规则 → 断言函数 → Layer 3 case → pytest 文件,每个断言标 covered / defined_unused / missing_implementation。它真的能抓问题:能发现某 oracle 引用了一个并不存在的断言函数(标 missing_implementation)——中立 demo-sut 自检即抓到 1 个这类教学缺口。bin/wave8-weekly 把证伪执行器与矩阵重生成串成一键周期入口。

纪律:质量分数只能来自真实运行——不提供结果文件时审计报告就是 not_run,不允许拍脑袋填数。这条纪律本身就是 第 5 条 在 Wave 8 上的体现:质量证据必须可证伪。
第 10 章 · 起源实证(Hypervise)

06-117 张真实订单证据进入 Wave 8 falsification,得分 86.7%。

这条 UC 证明了四件事,每件都对应前面章节的某个机制:① 门禁有牙齿(Wave 1 两次真实拒绝、物理门负向对照);② 找到真 bug(INC-2026-0610-01,第 5 条 兑现);③ 诚实归类(两个测试侧 oracle 误区被诚实归为测试侧错误,没有甩给开发——这正是 self-healing 的 human gate 精神);④ 拒绝假绿(候选池空时诚实 SKIP,没有为绿造数据)。一条 UC 同时跑通这四件事,是整套方法论"能落地、不是纸面"的活体证明。

规模化的诚实边界:全链路目前是 N=2——UC-LA-11(人工按阶段驱动)与 UC-AE-24(orchestrator --auto 驱动,2026-06-12)都已通过全量审计。两条把所有机制串通了,但"每个新 UC 都走 Wave 1 + 3B"还不是稳定的统计事实。距"Full SDTD 成熟"差的正是继续把这个规模化做厚(目标累计 3-5 条 UC),而非机制缺失。

第 12 章

挂载一个 SUT:profile 机制

早期 BUGate 与 Hypervise 耦合,开源的关键是把"随框架走的核心"与"按 SUT 重建的内容"分开。这一步现在已经完成:BUGate 是独立的 SUT 无关核心,一个 SUT 通过一份声明式 profile 绑定,曾经的 S1-S9 硬编码接缝已收敛进单一配置。

勘误 / 更新(2026-07-03, CHARTER-BUGATE-001):本章写作时以"SUT 测试工作区挂载进 BUGate 仓"为主叙事;定章后该布局仅为维护者工作台。默认形态是导入模式:SUT 测试仓为项目根,BUGate 被导入其中,profile 与 bugate.config.yaml 提交进 SUT 仓(同库同审),门禁跑在 SUT 仓 CI;根发现已拆分(工作区根=最近的 bugate.config.yaml,引擎资产按引擎自身位置解析)。四部分解耦模型不变,变的只是宿主方向。运行示例见 examples/imported-demo/(一等)与 examples/mounted-demo/(维护者)。

12.1 四部分解耦(回顾)

挂载关系是第 5.5 节的四部分:BUGate Core(本仓库,SUT 无关)→ SUT Profile(桥接契约)→ Mounted Workspace(通常即 SUT 的自动化测试框架 / 测试工作区)→ SUT / Product Runtime(黑盒目标)。核心纪律:挂载的是测试工作区,产品源码 / 密钥 / 环境事实不进入核心,Runtime 只作为黑盒证据被观测。

12.2 一份 profile 声明什么

核心默认 bugate.config.yaml 为 mode: core 且 guarded_path_regex: []——不守护任何路径,引擎可空跑。挂载时令 bugate.config.yaml 指向 profile: sut/<name>.profile.yaml(或设环境变量 BUGATE_PROFILE),profile 把下列 SUT 事实合并到核心之上:

字段作用
artifact_dir该 SUT 每个用例的 BUGate 产物目录(一 UC 一目录)
guarded_path_regex物理写入门守护的测试实现路径;空列表 = 不守护
required_precode_artifacts写守卫放行前必须 passed 的产物集(可加 1A/1B/2A)
agent_rolesWave 7 角色路径隔离(builder / designer / implementer 读写边界)
falsification_spec · falsification_thresholdWave 8 oracle falsification 的声明式 oracle / mutation 规格与分数门
prd_health_spec · prd_health_minWave 0 PRD 健康度自评规格与准入分
per-UC 硬化门require_multiview / require_adversarial_absorption / require_regression_traceability / layer2_strict / require_assertion_coverage / require_real_adversarial_dispatch / reject_on_bridge_failures(默认关,按 SUT 开启)
memory.namespace该 SUT 在记忆总线里的命名空间 / 标签

范本与字段全解见 examples/sample-sut.profile.yaml 与 .shared/skills/bugate/references/profile-schema.md。SUT profile 只能让核心不变量更严,不能更松。

12.3 自带的中立证据:demo-sut + 全自检

核心仓自带中立夹具,让任何人零配置就能看到门禁闭环:

  • examples/demo-sut/ —— 一个中立 URL 短链 SUT 的填满且通过的 01–05 门禁栈(含可选 1A/1B/2A)。python3 scripts/check_bugate_v13_semantics.py examples/demo-sut --scope all --require-passed 直接 PASS;其 Wave 0 自评 87.5 / A、Wave 8 证伪 85.7%。
  • examples/mounted-demo/ —— 一份挂载 profile 的走查:已 passed 的 UC 放行写入、pending 的 UC 被物理门 exit 2 阻断,演示"先过门再写代码"。
  • $bugate-full-check skill —— 一键端到端自检(四层门、双 runtime 真实互审、记忆总线、ONNX、Wave 0/8、写守卫、角色隔离),并明确区分"核心 + 可选运行时可用"与"真实 SUT 全部门禁已激活"。

另有 scripts/check_no_sut_terms.py(de-SUT 守卫)在 CI 中扫描核心的源码 / Markdown / YAML 等文本,确保 SUT 专属词汇不渗入引擎与方法论;它不覆盖 HTML 发布材料(如本白皮书),后者的中立性需人工把关。

12.4 对外口径(三句话)

  • 对开发:缺陷报告自带完整证据链(命题→oracle→case→断言→可观测证据),不拿测试侧推断当你的 bug。
  • 对测试同学:AI 干脏活,你保留所有否决权;每次纠正沉淀进记忆与规范,不会下周再犯。
  • 对采用者:SUT 无关核心 + 一份 profile 即可挂载;质量承诺可证伪——断言有效性有分数、门禁有拒绝记录、缺陷有闭环时间线。
第 13 章

前景展望:BUGate 的两条演进方向

BUGate 今天的实例化形态是"黑盒测试用例开发"。但它真正可复用的不是某条测试流水线,而是一组研发阶段无关的原语:需求评分、业务理解、多视角评审、门禁产物、物理写入门、记忆总线。这些原语并不依赖"被治理的对象是测试"——它们可以沿产品研发流程纵向前移到需求、设计、开发(TDD),并向白盒延伸。本章是设计上的可拓展预期(roadmap),不是既有能力声明。

定位升级:与其说 BUGate 是"一个 AI 黑盒测试框架",不如说它是一套"AI 辅助研发的治理底座"——把"先有据、再评审、过门禁、留记忆"做成机器可校验的纪律,而这套纪律对 PRD、设计、代码、测试一样成立。

13.1 为什么这些原语是阶段无关的

回看四类系统性失败(第 1 章):浅理解、假绿、臆造、知识蒸发——它们不是测试独有的,而是任何让 LLM 直接产出工件的环节都会犯的错。让 AI 写 PRD、写设计、写实现,同样会浅理解需求、自信臆造、留不下知识。因此 BUGate 的对策(命题化、证据分档、多视角分歧、门禁、记忆治理)天然适用于这些环节,只是"被治理的工件"从测试代码换成了 PRD / 设计 / 实现。

13.2 左移:需求与设计质量门

Wave 0 PRD 健康度评分本就与测试无关——它是一个声明式的需求质量打分器(8 维 + 可追溯性加成),输入 PRD 自评、输出 0–100 分 + A–D 等级 + 路由。放在研发流程最前端,PRD 一写完就评分,挡住"对烂 PRD 做精确实现/测试"。Layer 1 业务理解(命题 P-xxx、oracle O-xxx、域模型 1A、状态流 1B)是一份可被机器校验、可被人审阅的规格;多视角评审(两个不同厂商模型独立读同一输入、分歧即风险)可直接用于评审 PRD、设计文档、API 契约。三者合起来,就是一道开发开始前的需求/设计质量门——QA、PM、BA、开发都能用。

13.3 左移到开发:TDD 的规格层

TDD 的核心是"先写规格化的失败测试,再写实现"。BUGate 的 Layer 1 命题 + oracle 恰好就是开发者的可执行规格:每条命题带原文锚点与证据分档,每个 oracle 带可观测面。开发者(或开发侧 AI)据此先写失败测试、再写实现——而物理写入门这一机制是路径无关的:今天它守护测试实现路径(产物未 passed 禁写测试),同样可以配置成守护实现代码路径(规格未接受禁写实现),把"先过理解的门再写代码"直接变成 TDD 的红-绿纪律。这一步主要是一个 profile 变体 + 守护路径配置,产物与门禁机制都已存在。

13.4 向白盒延伸

BUGate 当前刻意走黑盒(第 9 章:黑盒语境下选 oracle falsification 而非白盒变异,有决策文档)。这是语境选择而非能力上限:一旦进入有源码访问的白盒语境,几条延伸是自然的——多视角评审可直接用于源码评审(两个模型独立读同一段实现,分歧即风险);Wave 8 可从"黑盒证据改错"扩展为对源码的变异测试;1A 域模型 / 1B 状态流可作为设计产物与实现做一致性核对。白盒与黑盒不冲突:同一套门禁/评审/记忆原语,黑盒看行为、白盒看实现。

13.5 记忆总线贯穿全生命周期

记忆总线(跨 agent SSOT + 治理门:无证据的 confirmed 拒写)今天沉淀的是测试发现与 incident。同一条总线可以贯穿 PRD 缺口、设计决策、契约约定、实现踩坑、线上 incident——成为研发全流程的单一可信知识库,让"知识蒸发"在每个环节都被结构性堵住,而不仅是测试环节。

13.7 另一条演进方向:自进化的治理内核(BUGate Loop)

前面 6 节是纵向拓展(覆盖更多研发阶段)。还有一条递归的方向:BUGate 本身就是一个语义治理内核,它可以从"治理测试开发"进化成"治理并改进自己的治理规则"——一个 evidence-driven(用真实执行证据)· human-ratified(经人审或元门禁接受)· profile-bounded(限定在 profile 与策略层) 的自我改进闭环。它不是"自己改自己然后自动信自己"。

BUGate 已经有这个闭环的胚胎(图 13-1):预编码产物 → 门禁(语义+物理) → 执行/证据 → 自愈+失败分类 → 04/05 → Memory Bus/晋升候选——这条链已经跑通。尚未"内核化"的是最后两步:把晋升候选制度化为治理提案,再让提案过一道元门禁。

图 13-1 BUGate 自进化治理闭环;※ 两步(治理提案 / 元门禁)尚未"内核化",经人审或元门禁接受后才晋升,回到下一轮(左侧蓝色箭头)

13.8 三层演进与不可逾越的边界

阶段做什么状态
Loop v0 · 学习闭环失败 / survived falsification / 人工纠错 / missing_implementation 自动进入 05_knowledge_update.md 与 Memory Bus,生成"晋升候选"。已有雏形
Loop v1 · 治理提案闭环新增 06_governance_proposal:把反复出现的问题转成候选规则——profile 加固、模板调整、新增 gate 检查、新增 de-SUT term、新增 oracle pattern。只提案,不自动改核心。待建
Loop v2 · 自进化内核原型每个治理提案跑元门禁:宪法检查 → SUT 中立性检查 → demo-sut 回归 → 对抗评审 → 向后兼容 → 人工接受。通过后才晋升为 core rule / profile schema / template / gate。研究级

这里的 demo-sut 回归 + 向后兼容就是治理的适应度函数:一条新规则必须"在历史证据上能多抓、不误伤,且不破坏既有",才配晋升——它把"治理质量"本身变成可证伪的指标,而不只是测量"断言质量"。

不可逾越的边界:自进化的对象只能是规则候选 / profile 加固 / 模板 / 检查项;绝不自动改五条宪法不变量,绝不把某个 SUT 的经验直接写进 core。否则会从治理内核退化成"自我强化偏见的规则泥团"(Goodhart:一个自我优化的系统会学会把门禁优化到形同虚设——这正是 BUGate 要防的假绿)。换句话说:内核(宪法 + 不变量检查 + 回归/元门禁)不参与进化,进化的只是它所治理的策略层。本章两条演进方向都是设计预期(roadmap)、半自治、人/元门禁批准;递归这条合起来可以叫 BUGate Loop: a self-evolving semantic governance kernel prototype。
第 14 章

结语

BUGate SDTD 的本质,是把"AI 写测试"从一种依赖资深经验的手艺,改造成一条任何 QA 都能可靠执行、可被机器校验、可被独立审计的工程流水线。

它的底层信念只有一条:当生成测试的是大语言模型时,诚实必须优先于绿色,而诚实不能靠提醒,只能靠机器可校验的产物与物理门禁来强制。这条信念贯穿了五条宪法、10 阶段流水线、双 runtime 互审、oracle falsification 质量证据,以及那个会拒绝无证据写入的记忆治理门。

它也是诚实的:起源期全链路 N=2(UC-AE-24 由 orchestrator --auto 端到端驱动,见第 10 章),self-healing 已在真实日志上运行、但"真实失败"分支尚未触发,这些都被如实标注为"成熟度问题"而非"机制缺失";白盒变异、行为 oracle 证伪等被明确划为"有意不做"的设计取舍。它的方法论根基是公认的 40 年测试研究,原创之处是面向 LLM 失败模式的重新编排——这个归因经得起开源社区的审视。

如果你打算用它:git clone 后先跑 check_bugate_v13_semantics.py examples/demo-sut --scope all --require-passed 看门禁全绿,再复制 examples/sample-sut.profile.yaml 把你的测试工作区挂上来,用 $bugate-full-check 验证安装。把对外口径定为"受治理的多 agent 黑盒测试开发",不要写"全自动";方法论的经典出处引用已在第 2 章列明,让它是"站在巨人肩膀上的严谨综合"而非"民间发明"。
