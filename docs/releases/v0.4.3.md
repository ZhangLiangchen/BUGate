# BUGate v0.4.3

[English](v0.4.3.md) | [简体中文](v0.4.3.zh-CN.md)

Status: final release notes for the v0.4.3 patch. Source implementation, test,
and documentation acceptance is complete. This source document does not claim
that main/tag CI, tag creation, asset publication, or public-download
reacceptance has completed; those remain release-operation gates.

BUGate v0.4.3 is an integrity patch for Wave 7 role evidence. It closes
the path where deleting `00_role_evidence/` could make an established UC look
new, and it makes lifecycle publication and recovery a journaled,
crash-recoverable saga across the workspace, a machine-level lineage registry,
and strict Memory checkpoints. It is not a distributed atomic transaction.
The durable defect record is
[`BUGATE-CORE-2026-07-23-ROLE-EVIDENCE-RESET`](../defects/BUGATE-CORE-2026-07-23-ROLE-EVIDENCE-RESET.md).

## Durable role-evidence lineage

Each governed UC has a deterministic identity key containing exactly:

```json
{
  "schema": "bugate.role-lineage-key/v1",
  "namespace": "<effective-memory-namespace>",
  "uc": "<resolved-uc-token>",
  "artifact_dir": "<canonical-workspace-relative-posix-path>"
}
```

`lineage_id` is the lowercase SHA-256 of the UTF-8 canonical JSON bytes: sorted
keys, compact separators, and no trailing newline. It contains no absolute
machine path, OS identity, credential, token, or SUT fact beyond the
profile-resolved UC identity and workspace-relative artifact location.

The local receipt directory is no longer the sole history-existence oracle.
v0.4.3 adds `role-lineage.sqlite3` under the effective machine Memory
home. It records the accepted head, sequence, lifecycle state, revision, Memory
mode, strict root/checkpoint IDs, and the sole active transaction. Read-only
paths do not create this registry. Registry schema version 2 adds
the durable initialization journal, exact next-stage graph enforcement, and
content-addressed root/checkpoint binding.

## Integrity state is separate from lifecycle state

`bugate-role status --json` and `lineage-status --json` report one of these
integrity states (the default human status line emphasizes lifecycle state):

| State | Meaning / operator route |
|---|---|
| `uninitialized` | No matching registry row and no local history. Confirm true first use, then run explicit `lineage-init`. |
| `aligned` | Registry and verified local chain agree on identity, head, sequence, lifecycle state, and Memory mode; no transaction is pending. |
| `migration_required` | Local history exists without a registry row, or explicit strict-root probing proves prior lineage existence. Adoption independently requires and verifies a non-empty valid legacy chain; otherwise review restoration. Never initialize over it. |
| `history_missing` | A registry row exists but all or part of its local chain is absent. |
| `history_diverged` | Local bytes/state or configured Memory mode disagree with the registered head. |
| `recovery_pending` | A durable initialization, publication, or recovery journal is incomplete. Inspect `active_initialization` versus `active_transaction` to select the resume command. |
| `registry_unavailable` | A present registry is unsafe/unreadable/schema-invalid, lineage context resolution fails, or an explicitly required root probe cannot be safely validated. A simply absent registry maps to `uninitialized` or `migration_required`. |

Only `aligned` admits a normal lifecycle publication. These values do not
replace lifecycle states such as `implementation_unlocked` or `closed`.

Ordinary `status`, hooks, and per-edit preflight remain local-only and make no
Memory HTTP request. `lineage-status` is an explicit operator diagnostic: only
when local state appears `uninitialized` under required Memory does it exact-GET
the deterministic root. If the root already exists while registry/local
history is absent, it returns non-zero `migration_required`. `lineage-init`
performs the same probe and refuses a false empty reset.

## Explicit first use, adoption, and recovery

The operator flow provides four commands:

```sh
# Read-only identity/integrity report. Non-zero is expected before first init.
.bugate/bin/bugate-role lineage-status docs/usecases/<UC> --json

# Genuine first use only; copy the exact lineage_id from lineage-status.
.bugate/bin/bugate-role lineage-init docs/usecases/<UC> \
  --lineage-id <exact-lineage-id>

# Existing verified pre-v0.4.3 history; no receipt byte is rewritten.
.bugate/bin/bugate-role lineage-adopt docs/usecases/<UC> \
  --lineage-id <exact-lineage-id> --expected-head <exact-chain-head>

# Registered missing/divergent history or an active publication/recovery transaction.
.bugate/bin/bugate-role recover docs/usecases/<UC> \
  --lineage-id <exact-lineage-id> --expected-head <exact-head-or-EMPTY> \
  [--archive <trusted-recovery-archive>]
```

`lineage-init` is a per-UC governance action and is unrelated to the
fresh-install-only `scripts/bugate_init.py`. `lineage-adopt` accepts only a
non-empty, fully verified legacy chain at the exact head and reports
`receipts_rewritten: 0`.

Initialization is not an unjournaled gap before the first registry row. Before
its first Memory request it persists one exact intent, then advances
`pending` -> `root_absence_verified` -> `root_verified` ->
`registry_initialized` -> `chain_written` -> `completed`. Required mode proves
root absence and creates/exact-verifies the deterministic root; best-effort
uses the same journal with an explicit no-remote-root boundary. Both modes
publish and exact-verify the mode-`0600` empty chain before completing. A crash
after intent creation reports `recovery_pending`; rerunning the same exact
`lineage-init` resumes the same stage. A strict root found at the initial probe
aborts the still-pre-root/pre-lineage intent and reports
`migration_required` instead.
Normal lifecycle publishers cannot bypass an active initialization.

Recovery validates the complete source and every existing target before
writing. An `active_initialization` resumes through exact `lineage-init`, not
`recover`. After validation and write preflight, `recover` selects the active
source or creates a pending `recovery_restore`, claims that exact source, and
only then restores the exact committed predecessor plus receipt/chain bytes and
modes. A live claimant is exclusive; a dead owner can be taken over only after
a liveness check and exact-token CAS. When an original lifecycle publication
has its exact receipt and verified checkpoint ready for CAS, recovery completes
that same transaction through CAS and local publication; it does not create a
duplicate next-sequence lifecycle receipt. In one SQLite transaction the
registry then terminalizes the claimed source (`recovery_restore` to `aborted`,
lifecycle to `completed`) and installs the sole canonical-bound, pending
`evidence_recovery` successor, so no `aligned`/no-audit crash gap exists. If the
active transaction is already `evidence_recovery`, retry resumes it directly
rather than appending another successor.
The recovery is visible; it is not a silent reset. A durable best-effort empty
transition ID is preserved as the unanchored marker without retrying Memory
HTTP.

## Transactional publication

Every lifecycle event now follows a durable sequence:

1. acquire the physical per-UC transition lock and require `aligned`;
2. create the sole registry transaction against the exact
   head/sequence/revision/checkpoint and bind the canonical transition before
   any Memory request;
3. in required mode, exact-verify the deterministic root and current committed
   predecessor checkpoint against the retained registry payload and exact
   head/sequence/state/revision;
4. prepare the transition Memory record according to
   `memory_mode`;
5. construct the receipt, finalize Memory against its receipt hash, and, in
   required mode, exact-verify the final transition/receipt binding;
6. only then freeze and bind the exact receipt bytes/path/mode/hash once in the
   registry and construct the exact minimal-chain bytes;
7. in required mode, publish and exact-verify an immutable checkpoint carrying
   the exact receipt and chain envelopes;
8. compare-and-swap the registry head;
9. publish the receipt with no-replace semantics and atomically replace
   `chain.json`; and
10. mark the transaction complete only after the local chain exists.

Predecessor exact verification is after the pending journal but before new
transition preparation. Checkpoint publication is after strict Memory
finalization and the exact registry receipt-byte bind, never before either one.

Any interruption after the durable lifecycle-publication journal is created is
detectable as `recovery_pending`; validation or handoff verification that fails
before that point creates no new pending transaction. Initialization has the
separate intent-first journal described above. A sole active transaction
serializes contenders before Memory work, and the registry CAS ensures that a
second workspace copy with the same deterministic lineage cannot win the same
head.

## Required versus best-effort Memory

- `memory_mode: required` creates and exact-verifies the deterministic Memory
  root and immutable checkpoint chain. Missing local evidence can be rebuilt
  from those checkpoints while the registry and Memory history survive. A
  separately trusted archive may also be supplied explicitly, but it selects
  candidate bytes only: retained strict checkpoints remain mandatory and
  authoritative, and every archive envelope must exactly match them before any
  write. It is not an offline fallback for unavailable or divergent strict
  Memory.
- `memory_mode: best_effort` still uses the machine registry for deletion
  detection and publication CAS and may attempt transition Memory calls while
  tolerating their failure, but creates no remote lineage root/checkpoint.
  Recovery after local-history loss therefore requires a separately retained,
  trusted `bugate.role-recovery-archive/v1` supplied with `--archive`.

Changing the configured Memory mode after lineage registration is integrity
drift, not a silent durability upgrade/downgrade.

## Updater and migration boundary

The imported updater may install the v0.4.3-capable engine, but it never edits
or creates a profile, namespace, `00_role_evidence/**`, the machine lineage
registry, or Memory data. It never runs `lineage-init`, `lineage-adopt`, or
`recover`.

A successful updater `apply`/`verify` therefore means only that the managed
engine projection is installed. It does **not** mean that existing UCs have
accepted lineage migration. After the engine update is reviewed and committed,
run `lineage-status` for each governed UC and perform the separately reviewed
exact init/adopt/recovery route its evidence requires.

The v0.4.2 tag, Release, and assets remain immutable. v0.4.3 is delivered as a
new patch release; the updater transports the new engine without rewriting the
previous release or silently migrating SUT-owned state.

## Security and limits

Hooks still cannot intercept arbitrary shell redirection, recursive deletion,
or an external editor. The registry plus strict Memory detect deletion and
drift; they do not authenticate the operator or provide non-repudiable
identity. Stronger controls require separate OS accounts, containers, managed
runners, protected backups, or role-scoped server credentials.

An actor with the same OS-user authority who deletes the workspace evidence,
the machine registry, and the complete Memory home has removed all local
anchors; that combined destruction is outside BUGate's local threat boundary.
Likewise, history lost before a deterministic registry/root was established
cannot be inferred from an empty directory. Restore trusted pre-loss evidence
or keep the migration blocked—never manufacture the missing past.

## Publication gates

The source-level lineage/Memory/hook/updater regressions, complete direct
discovery, de-SUT checks, bilingual documentation contract, and standalone
synthetic probes are recorded in the durable defect record. At the point these
source bytes are finalized, release-operation completion remains pending for
independent semantic reviews, clean archive-native `smoke + both`, merged-main
CI, annotated-tag CI, and public-download reacceptance. Publication must stop if
any gate fails.

The formal archive-native gate must run with the exact flags
`--full-check-mode smoke --full-check-archive both` so both tar and zip exercise
the lineage flow, updater preservation, rollback/reapply, and ownership oracles.
That smoke route deliberately interrupts strict Memory publication and resumes
it through `recover`; acceptance therefore binds the exact ordered six-event
lifecycle plus one state-preserving `evidence_recovery` receipt, for seven
exact anchors with lineage sequence/revision `0..6`. A stale six-count
release harness is a `NO-GO`, even when the archive runtime itself passed. The
release-infrastructure defect and regression are recorded in
[`BUGATE-CORE-2026-07-23-RELEASE-ACCEPTANCE-TRANSITION-COUNT`](../defects/BUGATE-CORE-2026-07-23-RELEASE-ACCEPTANCE-TRANSITION-COUNT.md).
Semantic release review remains provider-neutral and requires at least two real,
independent sessions. Cross-provider review is preferred; if one provider is
unavailable, only a newly spawned same-provider session with a different prompt
and no access to the first conclusion before synthesis is an acceptable
fallback. Placeholder output, a continued conversation, self-review, or an
optional provider outage relabeled as success never satisfies that gate.

The release distribution must contain exactly:

```text
bugate-0.4.3.tar.gz
bugate-0.4.3.zip
bugate-0.4.3.SHA256SUMS
```

Final SHA-256 values are generated only from the clean final commit and belong
in `bugate-0.4.3.SHA256SUMS` and the published GitHub Release record. This source
note intentionally embeds neither self-referential archive hashes nor a claim
that the assets already exist.
