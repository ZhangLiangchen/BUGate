# BUGate v0.4.2

[English](v0.4.2.md) | [简体中文](v0.4.2.zh-CN.md)

Release date: 2026-07-22

BUGate v0.4.2 preserves the Wave 7 governance contract introduced in v0.4.0
and corrected in v0.4.1, and adds the first-class incremental updater for
BUGate installations inside imported SUT test repositories. The updater is
SUT-neutral, stdlib-only, manifest-bound, fail-closed, transactional, and
reversible. It is not a second installer and it never turns an engine update
into an implicit governance migration.

The normative specification is the
[imported-mode updater contract](../qa-methodology/IMPORTED_UPDATER_CONTRACT.md).

## Wave 7 remains the governance boundary

This release does not weaken the existing pre-code or lifecycle gates:

- `designer` → `implementer` → `reviewer` remains an opt-in, fail-closed
  lifecycle controlled by the imported repository's profile.
- `00_role_evidence/`, strict Memory transition anchors, receipt hashes,
  handoff/session separation, drift re-locking, and the physical write guards
  retain their v0.4.1 semantics.
- Profiles without `role_governance`, and profiles with `mode: off`, keep the
  compatible legacy behavior. Installing a role-capable engine does not change
  that mode.
- Wave 1 peer review and Wave 7 lifecycle identity remain separate. The
  updater does not manufacture human acceptance, handoff, receipt, execution,
  or knowledge evidence.

## First-class imported-mode updater

Fresh installation and incremental update now have deliberately separate
entry points:

- `scripts/bugate_init.py` is for a new imported-mode installation. It creates
  the initial SUT-owned configuration/profile skeleton, discovery links, BUGate
  hook fragments, and installed lock. It refuses an existing supported
  installation instead of re-importing over it.
- An existing v0.3.x, v0.4.0, or v0.4.1 installation bootstraps once from an
  unpacked v0.4.2 release:

  ```sh
  cd <imported-sut-test-repository>
  python3 <unpacked-bugate-0.4.2>/scripts/bugate_update.py status .
  python3 <unpacked-bugate-0.4.2>/scripts/bugate_update.py plan .
  python3 <unpacked-bugate-0.4.2>/scripts/bugate_update.py apply .
  ```

- After v0.4.2 is installed, the stable vendored interface is:

  ```sh
  .bugate/bin/bugate-update status
  .bugate/bin/bugate-update plan --to 0.4.2
  .bugate/bin/bugate-update apply --to 0.4.2
  .bugate/bin/bugate-update verify
  .bugate/bin/bugate-update rollback --transaction <transaction-id>
  ```

Remote updates require an explicit SemVer target; there is no implicit
`latest`. Offline plan/apply accepts an archive together with its matching
`SHA256SUMS` asset. `status`, `plan`, `verify`, and `apply --dry-run` do not
persist target-repository state. A plan has a deterministic digest, and
`apply --plan` re-observes the complete base and rejects stale input.

## Ownership, release manifest, and installed lock

The generated release manifest is the complete authority for what the updater
may inspect and write. It distinguishes the full repository archive from the
smaller imported installation projection:

| Surface | Update rule |
|---|---|
| Vendored runtime | Only release-declared scripts, wrappers, shared BUGate skills, and runtime/setup documents under the configured vendor directory |
| Discovery and agents | Only release-declared BUGate skill links and Codex gate-agent files |
| Shared files | Only exact BUGate-owned hook entries in `.claude/settings.json` / `.codex/hooks.json` and the marked BUGate block in `.gitignore` |
| State | Deterministic installed metadata plus verified, Git-ignored transaction journals/backups/reports |
| SUT-owned content | Never managed: config/profile, tests, use cases, evidence, `AGENTS.md`, `CLAUDE.md`, non-BUGate hooks/skills, non-marked `.gitignore` text, Memory data, credentials, product facts, and environments |

Unknown files remain unknown; a managed directory is never recursively
replaced. A managed item whose current type, mode, target, or SHA-256 matches
neither its old nor new manifest is a conflict and makes the plan `NO-GO`.
There is no broad `--force` path.

Each release archive contains a canonical
`.bugate-release/manifest.json`. It inventories every archive entry and records
the exact `installed_projection`, archive roles, normalized type/mode/link/hash
data, layout/hook protocol versions, profile-schema compatibility, updater
floor, and a self-digest. The installed copy is
`<vendor-dir>/bugate.release.json`; `<vendor-dir>/bugate.lock.json` binds its
digest, the rendered projection, installed/previous versions, updater version,
and archive verification state. The lock is deterministic and contains no
absolute machine path, time, identity, credential, namespace, or SUT fact.

Remote/offline archive installation records the raw archive SHA-256. The
unpacked one-time bootstrap verifies the manifest and every mapped payload but
cannot reconstruct the original container digest, so the lock records
`unavailable-from-unpacked-source`; operators must verify the checksum before
extracting. A same-version no-op does not rewrite a valid lock merely because a
different archive container was supplied.

## Exact legacy adoption

The release builder derives and seals adoption manifests from these seven
formal annotated tags:

```text
v0.3.0  v0.3.1  v0.3.2  v0.3.4  v0.3.5  v0.4.0  v0.4.1
```

There was no v0.3.3 release. Support is defined by the seven shipped manifests,
their exact historical installed projections, modes, symlinks, marker blocks,
and hook shapes—not by a version string in an SUT document. A mixed fingerprint,
missing critical file, local managed modification, ambiguous layout, or
non-canonical legacy hook is `NO-GO`; the updater never guesses a “close enough”
source version.

## Transaction, rollback, and recovery

Before the first target write, the updater verifies the release source,
checksum where available, archive safety, manifest/version agreement, legacy
or installed baseline, profile compatibility, and the freshly observed plan.
It then acquires an exclusive workspace lock and:

1. persists a self-contained, verified transaction worker outside the vendor
   tree and snapshots every changing managed item, shared fragment, and lock;
2. stages and verifies new bytes, modes, directories, and symlink targets;
3. replaces only manifest-owned targets, safely deletes only unchanged stale
   owned items, and semantically merges owned hook/ignore fragments;
4. verifies the complete candidate installation before atomically publishing
   the new lock and committed report.

A handled failure or interrupt restores the pre-image. A crash leaves a durable
journal: read-only commands report `recovery_required`, while the next mutating
`apply` or explicit `rollback` performs fail-closed recovery. Explicit rollback
is itself locked, journaled, atomic, interrupt-safe, and crash-recoverable. It
restores only one committed transaction after proving the current owned state
still matches that transaction's post-image; a later update or local drift
makes rollback `NO-GO` instead of overwriting newer state.

Pre-lock bootstrap publication and archived rollback-state reuse use
same-filesystem atomic no-replace primitives on supported Darwin/Linux hosts.
If that guarantee is unavailable, apply fails before target writes. Concurrent
updaters, path exchange, symlink escape, descriptor/inode drift, journal/report
substitution, and partial publication all fail closed or restore through pinned
pre-images.

### Retained-history limit

Transaction history is intentionally capped at **128 entries**. Capacity is
reserved before normal/atomic preparation and before archive-rollback or
archived-bootstrap-reuse intent is published. At exactly 128 entries, every
public path that would need another entry rejects with zero persistent
target-repository byte or metadata change; an interrupted operation that brings
history to exactly 128 remains recoverable. The updater does not automatically
prune committed rollback evidence and exposes no public prune operation; a
store above 128 is invalid and fails closed. Operators must not claim unlimited
history or manually delete journals to bypass this boundary.

The v1 rollback image preserves logical file content/type/mode and symlink
targets, not extended attributes, ACLs, uid/gid, hardlink relationships,
timestamps, or inode identity. Back up any such external metadata dependency
before updating a managed/shared file.

## Profile, Memory, and activation boundaries

Engine update and governance activation are two different actions. The updater
may report `migration_available` for a compatible profile or
`migration_required` for an incompatible/unknown schema, but `plan` and `apply`
never edit `bugate.config.yaml`, `bugate.profile.yaml`,
`00_role_evidence/**`, or `memory.namespace`; they never switch
`role_governance.mode` to `required`.

The updater does not reconstruct, clear, migrate, or write Memory Bus data. A
successful engine transaction is not a strict-role transition. If strict role
governance is later enabled, its Memory-backed handoff/acceptance/completion
checks retain their existing fail-closed semantics.

Hook activation signals are evidence-based:

- Codex Desktop re-trust is required only when the update actually changes the
  bytes of `.codex/hooks.json`; the plan and result report
  `codex_hook_hash_changed`.
- Any changed Claude or Codex hook requires a new session before the new runtime
  surface is active; the plan/result reports `new_session_required`.
- Deliberately enabling Wave 7 still requires fresh, separate designer,
  implementer, and reviewer sessions. File/lock verification alone is not proof
  that hooks are active in an already-running Desktop process.

## Archive safety and security limits

The archive reader rejects absolute/traversing/duplicate/conflicting paths,
unsafe hardlinks or symlinks, link escapes, invalid modes or SemVer, ambiguous
or mismatched checksum records, and archive/manifest/plugin/updater/CLI version
disagreement before target writes. Archive metadata and Core/plugin extras are
readable only for validation unless the installed projection explicitly maps
them to a target.

SHA-256 plus an operator-chosen release archive is tamper-evident integrity
checking, not cryptographic publisher identity or a signed supply chain. It
does not prove who published a malicious but consistently checksummed asset.
Likewise, hook IDs, role environment variables, local receipts, and Memory
anchors provide auditability and drift detection—not non-repudiable identity.
Managed runners, OS/container isolation, or role-scoped server credentials are
still required for stronger identity and filesystem enforcement.

archive-native acceptance report as one CI audit artifact. It does **not**
publish a GitHub Release.

1. the exact clean-built tar and zip each pass archive-native extraction,
   bootstrap/update, imported smoke/full-check, same-version idempotence, and
   rollback-plus-verify acceptance while preserving non-owned fixtures;
2. the merged main commit and then the annotated `v0.4.2` tag each pass CI;
3. exactly the three assets below are attached manually to that tag; and
4. the public downloads are fetched again, SHA-256-verified, and smoke-tested.

## Distribution contract

The eventual GitHub Release must contain exactly:

```text
bugate-0.4.2.tar.gz
bugate-0.4.2.zip
bugate-0.4.2.SHA256SUMS
```

Download the checksum file with either archive and verify SHA-256 before
extraction. Prefer tar when symlink preservation matters. Development builds
using `--allow-dirty` or `--include-untracked` are previews and are never formal
release assets.
