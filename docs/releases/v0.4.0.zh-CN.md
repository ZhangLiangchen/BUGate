# BUGate v0.4.0

[English](v0.4.0.md) | [简体中文](v0.4.0.zh-CN.md)

> **已由 [v0.4.2](v0.4.2.zh-CN.md) 取代。** 本页继续作为 v0.4.0 的不可变
> 历史记录保留；其 tag 与已发布 assets 均不变。

发布日期：2026-07-20

BUGate v0.4.0 新增 opt-in、可审计的 Wave 7 生命周期角色治理；未启用该能力的
profile 继续保持 v0.3.x 行为。规范契约见
[Wave 7 角色治理协议](../qa-methodology/ROLE_GOVERNANCE_PROTOCOL.zh-CN.md)。

## 重点变化

- 新增 fail-closed 的 `designer` → `implementer` → `reviewer` 状态机，分别
  治理 pre-code、implementation 与 post-run。`bin/bugate-role` 提供
  `status`、`verify`、`approve`、`handoff`、`accept`、`complete`、
  `session-start` 与隔离的 `run` 命令。
- 每次转换都在 `<artifact-dir>/00_role_evidence/` 追加 hash-linked 本地
  receipt。Receipt 捕获 active profile 与相关工件 snapshot，并在对应边界加入
  实现文件及执行证据。profile、artifact、implementation、receipt 或 chain
  任一 drift，都会在受治理写入放行前被发现。
- `memory_mode: required` 使角色转换采用 strict Memory 写入与 exact-ID 读取。
  service unavailable、timeout、写失败、ID 不存在、namespace/role/UC/phase/hash
  不一致或 receipt 绑定失败都会非零退出，而且不发布本地解锁 receipt、不推进
  chain head。普通 recall、note 与 Stop heartbeat 仍会访问 Memory，但保持
  best-effort；每次编辑的角色检查只在本地执行，不发网络请求。
- orchestrator 与 Core artifact mutator 使用和 hook 相同的角色 preflight。
  `--auto` 输出 `READY_FOR_HUMAN_ACCEPTANCE`、
  `READY_FOR_DESIGNER_HANDOFF`、`IMPLEMENTATION_UNLOCKED`、
  `READY_FOR_REVIEWER_HANDOFF`、`POST_RUN_ACTIVE`、`CLOSED` 等生命周期
  状态；不会自动接受 03B。
- Claude 与 Codex hook 在原有 pre-code guard 之外新增
  `check_role_evidence.py`。agent tool 直接修改 `00_role_evidence/**` 一律拒绝。
  SessionStart 打印当前 mode/role/session；Stop 仍只是 best-effort heartbeat。
- Wave 1 peer 子进程会清除 `BUGATE_AGENT_ROLE`、`BUGATE_SESSION_ID` 以及
  receipt/session identity 变量，同时保留 proxy、model、effort、profile 与
  project-root 配置。Wave 1 peer 是独立设计评审 worker，不是 Wave 7 生命周期角色。
- config loader 现在用确定性的 deep merge 保留 nested mapping：mapping 递归合并，
  profile scalar 覆盖 base scalar，profile list 整体替换 base list；继续兼容
  `memory.namespace`/`namespace` 与 `bugate.mode/version`/顶层
  `mode/version` 这两组 v0.3.x targeted alias，不恢复通用 nested flatten。

## 状态机

| 事件 | 必须满足的边界 | 结果 |
|---|---|---|
| `human_acceptance` | designer 仅记录已经发生的人工决定；配置要求的 03B 已是 `gate_status: passed` | `ready_for_designer_handoff` |
| `designer_handoff` | 当前 pre-code/provenance snapshot 加 strict Memory 锚点 | `awaiting_implementer_acceptance` |
| `implementer_acceptance` | 不同 implementer session 接受 exact handoff Memory ID | `implementation_unlocked` |
| `implementer_handoff` | hash snapshot 至少一个命中 guard 且属于同一 UC 的实现文件 | `awaiting_reviewer_acceptance` |
| `reviewer_acceptance` | 不同 reviewer session 接受 exact handoff 与当前实现 snapshot | `post_run_active` |
| `reviewer_completion` | 记录 04/05、命令/exit code、执行证据 hash 与最终 gate | passed 时进入 `closed` |

Receipt 使用 `bugate.role-evidence/v1`，`chain.json` 使用
`bugate.role-chain/v1`。JSON 采用 UTF-8、sorted keys、紧凑 separators；计算
receipt hash 时排除 `receipt_sha256`。发布过程使用同目录临时文件、`fsync`
与 `os.replace`。

## 兼容与迁移

- 不含 `role_governance` 或设为 `role_governance.mode: off` 的 profile 保持
  v0.3.x 行为。Core 默认仍是 `mode: off`，不会自锁。
- `agent_roles` 仍是独立的路径读写策略，可以单独使用；不能与
  `role_governance` 合并成一个含义模糊的配置块。
- 启用 `mode: required` 后，角色未设置/错误或缺 required session ID 都会硬阻塞。
  历史 passed UC 不会被 grandfather：Layer 4 前必须新建当前 human acceptance、
  designer handoff 与 implementer acceptance；post-run 同样需要 implementer
  handoff 与 reviewer acceptance。
- rerun `bugate_init.py` 可刷新 vendored scripts 与 BUGate-owned hook，同时保留
  SUT-owned hook。designer、implementer、reviewer 必须用三个新 session。hook
  子进程不能向父进程 export 变量；已运行的 Desktop app 也不会自动获得 shell
  环境变更，必须从目标环境重新启动。
- 人工把 03B 设为 passed 后，不要再执行 `--auto`。直接用
  `bugate-role approve` 记录既有决定，再执行 handoff。
- Memory 故障只阻塞下一次 strict 角色转换，不拖垮每次编辑；恢复服务后幂等
  重试。profile/pre-code drift 需要新 human/designer generation；实现 drift
  需要新 implementer handoff 与 reviewer acceptance。禁止删除或手改 evidence
  来 reset；receipt 被篡改时先从可信证据恢复，再追加 superseding transition。

## 发布资产与校验

GitHub Release 只包含三个资产：

```text
bugate-0.4.0.tar.gz
bugate-0.4.0.zip
bugate-0.4.0.SHA256SUMS
```

必须随 archive 一起下载 checksum 文件，并在解压前校验 SHA-256。需要可靠保留
symlink 时优先 tar；zip 包含等价 kit。

## 安全与激活边界

`approved_by` 只是“既有人类决定”的声明记录，不是身份认证。环境角色、本地
hook、本地文件、hash 链接与外部 Memory 锚点能提供可审计性与篡改/drift 检测，
但不能提供不可抵赖身份。强身份隔离需要独立 OS 账号、容器、managed runner
或按角色发放的服务端凭据。

Hook 无法拦截任意 shell 重定向或外部编辑器写入。受支持的 agent tool、
orchestrator 与 Core artifact mutator 会被强制治理；更强的文件系统控制属于
managed runner。最后，本版本改变了 Codex hook hash；导入仓在 Codex Desktop
中显式 re-trust 前，hook 可能静默不生效。文件/re-vendor 验收通过不等于 runtime
enforcement 已激活。
