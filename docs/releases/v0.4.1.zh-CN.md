BUGate v0.4.1 是一个 Core full-check fixture 隔离缺陷的补丁版本。该缺陷在使用
已经发布的 v0.4.0 archive 对 imported workspace 做验收时发现。v0.4.0 tag 与
assets 保持不可变，不会被重新打 tag 或替换。durable defect 记录见
[`BUGATE-CORE-2026-07-20-FULL-CHECK-FIXTURE-PROFILE-LEAK`](../defects/BUGATE-CORE-2026-07-20-FULL-CHECK-FIXTURE-PROFILE-LEAK.md)。

## 修复：imported full-check synthetic profile 泄漏

在 v0.4.0 中，Core smoke 可以通过；但当 imported workspace 的 active profile
启用 `role_governance.mode: required` 时，synthetic multiview hardening fixture
可能在初始化前失败。临时 UC 位于 imported workspace 之外，而 init 命令继承了
调用方的 required profile。required role preflight 正确拒绝了这个 workspace 外路径；
缺陷位于 full-check fixture 边界，不在 SUT，也不在 gate。

v0.4.1 将所有 synthetic Core-template、guard、hardening、dialect 与 real-peer
probe 和调用方的 SUT 配置隔离：

- fixture mutation 之前先创建 probe 自有 profile，并通过 `BUGATE_PROFILE` 传给
  所有消费该 fixture 的命令。
- probe profile 显式设置 `role_governance.mode: off`，绑定临时 artifact root，
  并使用专用 `project:bugate-full-check` Memory namespace。
- `BUGATE_PROJECT_ROOT`、进程 CWD、profile 与 Memory namespace 全部指向同一个
  临时 probe root；外层 imported workspace 保持不变。
- hardening control 分别报告 init、baseline semantic PASS，以及精确的
  missing-`divergence_report` negative result。
- full mode 的 init、multiview 与 adversarial dispatch 共用一个隔离的
  profile/root/CWD/Memory 环境。

Codex 默认拒绝 non-git CWD，因此两个 peer bridge 新增显式、默认关闭的自动化开关
`SDTD_CODEX_SKIP_GIT_REPO_CHECK=1`。只有 synthetic full-check real-peer probe
会启用它。普通 imported peer dispatch 继续保留 Codex repository check，Codex 仍在
`read-only` sandbox mode 下运行，hook trust 也不会被绕过。

## 回归控制

- imported-layout 回归提供一个外层 required role profile，先证明旧的未隔离 init
  会被拒绝，再要求隔离后的 hardening probe 在没有 lifecycle identity、且不修改外层
  workspace 的情况下通过。
- peer-environment 回归要求 init、multiview 与 adversarial 命令共用同一个临时
  profile、root、CWD、Memory namespace 与显式 non-git opt-in。
- peer-role 回归继续证明 lifecycle identity 会被清除，且新的 Codex 选项除非显式
  选择，否则保持关闭。

## 不变的契约与兼容性

- semantic、pre-code、role、handoff 与 post-run gate 均未弱化。
- `bugate.role-evidence/v1`、`bugate.role-chain/v1`、receipt hashing、strict
  Memory transition 与 drift 重锁均未改变。
- 不含 `role_governance` 的 profile 与使用 `mode: off` 的 profile 继续保持
  v0.3.x-compatible 行为；Core 默认仍保持 inert。
- SUT 自有 hooks 与 profile 继续归 imported repo 所有。重新执行
  `bugate_init.py` 即可刷新 vendored full-check 与 bridge scripts。

安全边界也没有改变。角色声明、session 区分、hash、hooks 与 Memory anchor 提供的是
可审计性，不是不可抵赖身份。强身份仍需要 OS/container 隔离、managed runner 或按角色
发放的服务端凭据。Hooks 不能拦截任意 shell redirection 或 external-editor write。

## Codex hook trust

本补丁不修改 `.codex/hooks.json`、`.claude/settings.json` 或 plugin hook commands，
因此从已信任的 v0.4.0 升级不会再次改变 hook hash。尚未信任 v0.4.0 Wave 7 hook
更新的操作者仍须在 Codex Desktop 中显式 re-trust；仅验证文件或 re-vendor 不能证明
runtime enforcement 已激活。

## 分发与校验

GitHub Release 只包含三个 assets：

```text
bugate-0.4.1.tar.gz
bugate-0.4.1.zip
bugate-0.4.1.SHA256SUMS
```

必须随任一 archive 一起下载 checksum 文件，并在解压前验证 SHA-256。正式验收要求
从两个解压后的 archive 分别执行 smoke check，再使用下载 release 完成 scratch
imported dry-run/init 与 imported smoke。release builder 默认拒绝 dirty tree；开发态
`--allow-dirty` 与 `--include-untracked` 不能用于正式 assets。
