# BUGate v0.4.2

[English](v0.4.2.md) | [简体中文](v0.4.2.zh-CN.md)

发布日期：2026-07-22

BUGate v0.4.2 保留 v0.4.0 引入、v0.4.1 修正的 Wave 7 治理契约，并首次为
imported SUT 测试仓内的 BUGate 安装提供一等增量更新器。更新器保持 SUT-neutral、
stdlib-only、manifest-bound、fail-closed、transactional 且可回滚；它不是第二套
installer，也不会把 engine update 偷换成隐式的治理迁移。

规范定义见
[imported-mode updater contract](../qa-methodology/IMPORTED_UPDATER_CONTRACT.zh-CN.md)。

## Wave 7 继续作为治理边界

本版本不弱化已有 pre-code 或 lifecycle gate：

- `designer` → `implementer` → `reviewer` 仍是由 imported repo profile 控制、
  opt-in 且 fail-closed 的生命周期。
- `00_role_evidence/`、strict Memory transition anchor、receipt hash、handoff/session
  隔离、drift 重锁与物理写入 guard 保持 v0.4.1 语义。
- 不含 `role_governance` 的 profile，以及使用 `mode: off` 的 profile，继续保持
  兼容的 legacy 行为。安装具备角色治理能力的 engine 不会改变该 mode。
- Wave 1 peer review 与 Wave 7 lifecycle identity 继续分离。更新器不会制造
  human acceptance、handoff、receipt、execution 或 knowledge evidence。

## 一等 imported-mode 更新器

fresh install 与 incremental update 现在使用刻意分离的入口：

- `scripts/bugate_init.py` 只负责新的 imported-mode 安装：创建首份 SUT-owned
  config/profile skeleton、discovery link、BUGate hook fragment 与 installed lock。
  遇到受支持的已有安装时，它会拒绝覆盖式 re-import。
- 已有 v0.3.x、v0.4.0 或 v0.4.1 安装，使用解压后的 v0.4.2 release 完成一次性
  bootstrap：

  ```sh
  cd <imported-sut-test-repository>
  python3 <unpacked-bugate-0.4.2>/scripts/bugate_update.py status .
  python3 <unpacked-bugate-0.4.2>/scripts/bugate_update.py plan .
  python3 <unpacked-bugate-0.4.2>/scripts/bugate_update.py apply .
  ```

- 安装 v0.4.2 后，稳定的 vendored 入口是：

  ```sh
  .bugate/bin/bugate-update status
  .bugate/bin/bugate-update plan --to 0.4.2
  .bugate/bin/bugate-update apply --to 0.4.2
  .bugate/bin/bugate-update verify
  .bugate/bin/bugate-update rollback --transaction <transaction-id>
  ```

远程更新必须显式给出合法 SemVer target，不存在隐式 `latest`。离线 plan/apply
同时接受 archive 与匹配的 `SHA256SUMS` asset。`status`、`plan`、`verify` 与
`apply --dry-run` 不会向目标仓持久化状态。Plan 带有确定性 digest；
`apply --plan` 会重新观察完整 base，并拒绝 stale input。

## Ownership、release manifest 与 installed lock

生成的 release manifest 是更新器可检查/可写范围的完整权威来源，并把完整仓库
archive 与更窄的 imported installation projection 分开：

| 范围 | 更新规则 |
|---|---|
| Vendored runtime | 只管理 release 声明的 scripts、wrappers、共享 BUGate skills，以及 configured vendor directory 下的 runtime/setup 文档 |
| Discovery 与 agents | 只管理 release 声明的 BUGate skill links 和 Codex gate-agent 文件 |
| Shared files | 只管理 `.claude/settings.json` / `.codex/hooks.json` 中精确的 BUGate-owned hook entries，以及 `.gitignore` 中有 marker 的 BUGate block |
| State | 确定性的 installed metadata，以及经过验证且被 Git ignore 的 transaction journal/backup/report |
| SUT-owned content | 永不管理：config/profile、tests、use cases、evidence、`AGENTS.md`、`CLAUDE.md`、非 BUGate hook/skill、非 marker `.gitignore` 文本、Memory data、凭据、产品事实与环境 |

未知文件继续保持未知；更新算法不会递归替换 managed directory。若某个 managed
item 当前的 type、mode、target 或 SHA-256 既不匹配旧 manifest，也不匹配新
manifest，plan 就会因 conflict 而 `NO-GO`。不存在宽泛的 `--force` 路径。

每个 release archive 都带 canonical `.bugate-release/manifest.json`，其中完整
记录所有 archive entry、精确 `installed_projection`、archive role、规范化的
type/mode/link/hash、layout/hook protocol version、profile schema compatibility、
updater floor 与 self-digest。安装后的副本是
`<vendor-dir>/bugate.release.json`；`<vendor-dir>/bugate.lock.json` 绑定其 digest、
渲染后的 projection、当前/上一版本、updater version 与 archive verification
状态。Lock 是确定性的，不含绝对机器路径、时间、身份、凭据、namespace 或 SUT 事实。

远程/离线 archive 安装会记录 raw archive SHA-256。一次性 unpacked bootstrap
会验证 manifest 与每个 mapped payload，但无法还原原始 container digest，因此
lock 记录 `unavailable-from-unpacked-source`；operator 必须在解压前校验 checksum。
同版本 no-op 不会仅因换用了另一个 archive container 而重写有效 lock。

## 精确 legacy adoption

Release builder 从以下七个正式 annotated tags 生成并封存 adoption manifests：

```text
v0.3.0  v0.3.1  v0.3.2  v0.3.4  v0.3.5  v0.4.0  v0.4.1
```

不存在 v0.3.3 release。支持范围由这七份随包 manifest 以及其精确的历史 installed
projection、mode、symlink、marker block 与 hook shape 定义，而不是由 SUT 文档中的
版本字符串定义。mixed fingerprint、关键文件缺失、local managed modification、
ambiguous layout 或非 canonical legacy hook 都会 `NO-GO`；更新器不会猜测一个
“足够接近”的来源版本。

## Transaction、rollback 与 recovery

第一次写目标仓之前，更新器会校验 release source、可用时的 checksum、archive
safety、manifest/version 一致性、legacy 或 installed baseline、profile
compatibility，以及刚刚重新观察的 plan。随后获取 exclusive workspace lock，并：

1. 在 vendor tree 外持久化一套 self-contained、已验证的 transaction worker，
   snapshot 每个要变化的 managed item、shared fragment 与 lock；
2. stage 并验证新 bytes、mode、directory 与 symlink target；
3. 只替换 manifest-owned target，只安全删除仍与旧基线一致的 stale owned item，
   并对 owned hook/ignore fragment 做 semantic merge；
4. 完整验证候选安装后，再原子发布新 lock 与 committed report。

可处理的 failure 或 interrupt 会恢复 pre-image。Crash 会留下 durable journal：
read-only command 只报告 `recovery_required`，下一次 mutating `apply` 或显式
`rollback` 才执行 fail-closed recovery。显式 rollback 自身也会加锁、写 journal，
并保持 atomic、interrupt-safe、crash-recoverable。它只在当前 owned state 仍精确
匹配目标 transaction post-image 时恢复该 transaction；后续 update 或 local drift
会让 rollback `NO-GO`，而不是覆盖更新状态。

Pre-lock bootstrap 发布与 archived rollback-state reuse 在受支持的 Darwin/Linux
主机上使用同文件系统 atomic no-replace primitive；若平台无法保证这一点，apply
会在目标写入前失败。concurrent updater、path exchange、symlink escape、
descriptor/inode drift、journal/report substitution 与 partial publication 均会
fail closed，或通过 pinned pre-image 恢复。

### Retained-history 上限

Transaction history 有意限制为 **128 entries**。正常/atomic prepare，以及发布
archive-rollback 或 archived-bootstrap-reuse intent 之前都会先预留容量。正好已有
128 entries 时，所有还需新增 entry 的 public path 都会拒绝，并保持目标仓
byte/metadata 零持久化变化；使 history 正好达到 128 的 interrupted operation 仍然
可以 recovery。更新器不会自动 prune 已提交 rollback evidence，也不提供 public
prune operation；超过 128 的 store 属于非法状态并 fail closed。Operator 不得宣称
history 无上限，也不得手工删除 journal 来绕过边界。

v1 rollback image 保证 logical file content/type/mode 与 symlink target，不保证
extended attributes、ACL、uid/gid、hardlink relationship、timestamp 或 inode
identity。若 managed/shared file 依赖这些外部 metadata，更新前必须独立备份。

## Profile、Memory 与激活边界

Engine update 与治理激活是两个动作。更新器可以针对兼容 profile 报告
`migration_available`，针对不兼容/未知 schema 报告 `migration_required`；但
`plan` 与 `apply` 永不编辑 `bugate.config.yaml`、`bugate.profile.yaml`、
`00_role_evidence/**` 或 `memory.namespace`，也不会把
`role_governance.mode` 切换为 `required`。

更新器不会重建、清空、迁移或写入 Memory Bus data。成功的 engine transaction
不是 strict-role transition。若以后启用 strict role governance，其 Memory-backed
handoff/acceptance/completion check 继续保持原有 fail-closed 语义。

Hook 激活信号基于实际 evidence：

- 只有更新确实改变 `.codex/hooks.json` bytes 时，Codex Desktop 才需要 re-trust；
  plan/result 会报告 `codex_hook_hash_changed`。
- 任一 Claude 或 Codex hook 发生变化后，都要新建 session 才能启用新 runtime
  surface；plan/result 会报告 `new_session_required`。
- 主动启用 Wave 7 时，仍必须分别启动全新的 designer、implementer、reviewer
  session。仅通过 file/lock verification 不能证明已运行 Desktop process 中的
  hooks 已激活。

## Archive safety 与安全边界

Archive reader 会在目标写入前拒绝 absolute/traversing/duplicate/conflicting path、
unsafe hardlink 或 symlink、link escape、非法 mode 或 SemVer、ambiguous/mismatched
checksum record，以及 archive/manifest/plugin/updater/CLI version 不一致。Archive
metadata 与 Core/plugin extra 只可用于读取校验；只有 installed projection 显式映射
到 target 的内容才可安装。

SHA-256 加 operator 选择的 release archive 只提供 tamper-evident integrity check，
不是 cryptographic publisher identity 或 signed supply chain；它不能证明是谁发布了
一份恶意但 checksum 自洽的 asset。同样，hook ID、role 环境变量、本地 receipt 与
Memory anchor 提供的是可审计性和 drift detection，不是不可抵赖身份。更强的身份与
文件系统治理仍需要 managed runner、OS/container isolation 或 role-scoped server
credential。

## 验证证据与发布门禁

Transaction-integrity 缺陷与 closure evidence 记录在
[`BUGATE-CORE-2026-07-21-UPDATER-TRANSACTION-INTEGRITY`](../defects/BUGATE-CORE-2026-07-21-UPDATER-TRANSACTION-INTEGRITY.md)。
独立关闭的 release-gate false-green controls 记录在
[`BUGATE-CORE-2026-07-22-RELEASE-ACCEPTANCE-INTEGRITY`](../defects/BUGATE-CORE-2026-07-22-RELEASE-ACCEPTANCE-INTEGRITY.md)。
独立复核的 pre-release baseline 报告：

- focused transaction suite：**93/93 PASS**；
- focused archive-native acceptance contract：**12/12 PASS**；
- complete direct unittest discovery：**349/349 PASS**；
- compileall、四个 template semantic gates、两组 de-SUT reading、de-SUT
  meta-test、full-check layout acceptance 与 `git diff --check`：PASS；
- exact-128 零写入/recovery probes：normal apply、atomic prepare、archiving
  rollback、archived bootstrap reuse 与 interrupted recovery 全部 PASS。

CI 现在会拉取完整 Git history/tags，验证七个 legacy ref 都是 annotated tag 且能
peel 到 commit，强制 repository/plugin/updater/current-tag version 一致，运行已有
全部 gate/test，从 clean checkout 构建真实仓库 tar/zip，校验 checksum 文件，并将
三个已验证 release assets 与 machine-readable archive-native acceptance report
共同保留为一个 CI audit artifact。CI **不会**发布 GitHub Release。

在以下后续门禁全部有证据之前，publication 继续 `NO-GO`：

1. 精确 clean-built tar 与 zip 都通过 archive-native extraction、bootstrap/update、
   imported smoke/full-check、same-version idempotence、rollback-plus-verify 验收，
   且非 owned fixture 保持不变；
2. merged main commit 与随后创建的 annotated `v0.4.2` tag 分别通过 CI；
3. 下列且仅下列三个 assets 被人工附到该 tag；
4. 再次下载 public assets，完成 SHA-256 校验与 smoke test。

## 分发契约

最终 GitHub Release 必须且只能包含：

```text
bugate-0.4.2.tar.gz
bugate-0.4.2.zip
bugate-0.4.2.SHA256SUMS
```

必须随任一 archive 下载 checksum 文件，并在解压前验证 SHA-256。需要保留 symlink
时优先使用 tar。使用 `--allow-dirty` 或 `--include-untracked` 生成的 development
build 只是 preview，绝不能作为正式 release asset。
