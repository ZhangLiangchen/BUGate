# BUGate v0.4.1

[English](v0.4.1.md) | [简体中文](v0.4.1.zh-CN.md)

Release date: 2026-07-20

BUGate v0.4.1 is a patch release for a Core full-check fixture-isolation defect
discovered while accepting the already-published v0.4.0 archive in an imported
workspace. The v0.4.0 tag and assets remain immutable; they are not retagged or
replaced. The durable defect record is
[`BUGATE-CORE-2026-07-20-FULL-CHECK-FIXTURE-PROFILE-LEAK`](../defects/BUGATE-CORE-2026-07-20-FULL-CHECK-FIXTURE-PROFILE-LEAK.md).

## Fixed: imported full-check synthetic profile leakage

In v0.4.0, Core smoke passed, but an imported workspace whose active profile
enabled `role_governance.mode: required` could make the synthetic multiview
hardening fixture fail before initialization. The temporary UC lived outside
the imported workspace, while its init command inherited the caller's required
profile. Required role preflight correctly rejected that out-of-workspace path;
the defect was in the full-check fixture boundary, not the SUT or the gate.

v0.4.1 isolates every synthetic Core-template, guard, hardening, dialect, and
real-peer probe from the caller's SUT configuration:

- A probe-owned profile is created before fixture mutation and passed through
  `BUGATE_PROFILE` to every command that consumes that fixture.
- The probe profile explicitly sets `role_governance.mode: off`, binds its
  temporary artifact root, and uses the dedicated
  `project:bugate-full-check` Memory namespace.
- `BUGATE_PROJECT_ROOT`, process CWD, profile, and Memory namespace all point at
  the same temporary probe root. The outer imported workspace remains unchanged.
- The hardening control separately reports init, baseline semantic PASS, and
  the precise missing-`divergence_report` negative result.
- Full-mode init, multiview, and adversarial dispatch share one isolated
  profile/root/CWD/Memory environment.

Codex normally rejects a non-git CWD. Both peer bridges therefore expose
`SDTD_CODEX_SKIP_GIT_REPO_CHECK=1` as an explicit, default-off automation flag.
Only the synthetic full-check real-peer probe enables it. Normal imported peer
dispatch retains Codex's repository check, Codex still runs in `read-only`
sandbox mode, and hook trust is not bypassed.

## Regression controls

- An imported-layout regression supplies an outer required role profile, proves
  the old unisolated init would be rejected, then requires the isolated
  hardening probe to pass without lifecycle identity or workspace mutation.
- A peer-environment regression requires init, multiview, and adversarial
  commands to share the same temporary profile, root, CWD, Memory namespace,
  and explicit non-git opt-in.
- The peer-role regression continues to prove lifecycle identity stripping and
  that the new Codex option is off unless explicitly selected.

## Unchanged contracts and compatibility

- No semantic, pre-code, role, handoff, or post-run gate was weakened.
- `bugate.role-evidence/v1`, `bugate.role-chain/v1`, receipt hashing, strict
  Memory transitions, and drift re-locking are unchanged.
- Profiles without `role_governance` and profiles using `mode: off` retain
  v0.3.x-compatible behavior. Core remains inert by default.
- Existing SUT-owned hooks and profiles remain owned by the imported repo.
  Re-run `bugate_init.py` to refresh the vendored full-check and bridge scripts.

The security boundary is also unchanged. Role declarations, session separation,
hashes, hooks, and Memory anchors provide auditability, not non-repudiable
identity. Strong identity still requires OS/container isolation, managed
runners, or role-scoped server credentials. Hooks cannot intercept arbitrary
shell redirection or external-editor writes.

## Codex hook trust

This patch does not change `.codex/hooks.json`, `.claude/settings.json`, or the
plugin hook commands, so upgrading an already trusted v0.4.0 installation does
not introduce another hook hash change. Operators who never trusted the v0.4.0
Wave 7 hook update must still explicitly re-trust it in Codex Desktop; file or
re-vendor verification alone is not proof that runtime enforcement is active.

## Distribution and verification

The GitHub Release contains exactly three assets:

```text
bugate-0.4.1.tar.gz
bugate-0.4.1.zip
bugate-0.4.1.SHA256SUMS
```

Download the checksum file with either archive and verify SHA-256 before
extraction. Formal acceptance requires smoke checks from both extracted
archives, followed by scratch imported dry-run/init and imported smoke using the
downloaded release. The release builder rejects a dirty tree by default;
development-only `--allow-dirty` and `--include-untracked` are not valid for
formal assets.
