## 持久 role-evidence lineage

每个受治理 UC 都有一份确定性 identity key，且只包含：

```json
{
  "schema": "bugate.role-lineage-key/v1",
  "namespace": "<effective-memory-namespace>",
  "uc": "<resolved-uc-token>",
  "artifact_dir": "<canonical-workspace-relative-posix-path>"
}
```

`lineage_id` 是这份 UTF-8 canonical JSON bytes 的小写 SHA-256：key 排序、紧凑
separator、无结尾换行。它不包含 absolute machine path、OS identity、credential、
token，也不包含 profile 解析出的 UC identity 与 workspace-relative artifact location
之外的 SUT 事实。

本地 receipt 目录不再是唯一的“历史是否存在”判据。v0.4.3 在 effective machine
Memory home 下新增 `role-lineage.sqlite3`，记录已经接受的 head、sequence、lifecycle
state、revision、Memory mode、strict root/checkpoint ID 与唯一 active transaction。
Read-only path 不会创建这份 registry。Registry schema version 2 新增
durable initialization journal、exact next-stage graph enforcement 与
content-addressed root/checkpoint binding。

## Integrity state 与 lifecycle state 分离

`bugate-role status --json` 与 `lineage-status --json` 会报告以下 integrity state 之一
（默认 human status line 重点显示 lifecycle state）：

| 状态 | 含义 / operator 路径 |
|---|---|
| `uninitialized` | 没有匹配 registry row，也没有本地历史。确认确属首次使用后，显式执行 `lineage-init`。 |
| `aligned` | Registry 与已验证本地 chain 在 identity、head、sequence、lifecycle state、Memory mode 上一致，且无 pending transaction。 |
| `migration_required` | 存在本地历史但尚无 registry row，或显式 strict-root probe 证明曾存在 lineage。Adoption 会独立要求并验证非空有效 legacy chain；否则审查 restore。禁止覆盖式 init。 |
| `history_missing` | Registry row 存在，但本地 chain 全部或部分缺失。 |
| `history_diverged` | 本地 bytes/state 或配置的 Memory mode 与 registered head 不一致。 |
| `recovery_pending` | Durable initialization、publication 或 recovery journal 尚未完成。检查 `active_initialization` 与 `active_transaction` 后选择对应的继续命令。 |
| `registry_unavailable` | 已存在 registry 不安全/不可读/schema 非法、lineage context 解析失败，或显式要求的 root probe 无法安全校验。单纯 registry 缺失会映射为 `uninitialized` 或 `migration_required`。 |

只有 `aligned` 才能发布普通 lifecycle event。这些状态不会取代
`implementation_unlocked`、`closed` 等 lifecycle state。

普通 `status`、hooks 与 per-edit preflight 继续保持 local-only，不发 Memory HTTP
请求。`lineage-status` 是显式 operator diagnostic：只有 required Memory 下本地状态
看似 `uninitialized` 时，它才 exact-GET 确定性 root。如果 registry/local history
缺失而 root 已存在，它会非零返回 `migration_required`。`lineage-init` 做同样 probe，
并拒绝伪造空白 reset。

## 显式 first use、adoption 与 recovery

Operator flow 提供四个命令：

```sh
# 只读 identity/integrity 报告；首次 init 前非零是预期结果。
.bugate/bin/bugate-role lineage-status docs/usecases/<UC> --json

# 仅限真实首次使用；从 lineage-status 复制 exact lineage_id。
.bugate/bin/bugate-role lineage-init docs/usecases/<UC> \
  --lineage-id <exact-lineage-id>

# 已有且通过验证的 pre-v0.4.3 历史；不重写任何 receipt byte。
.bugate/bin/bugate-role lineage-adopt docs/usecases/<UC> \
  --lineage-id <exact-lineage-id> --expected-head <exact-chain-head>

# 已注册但缺失/分歧的历史，或 active publication/recovery transaction。
.bugate/bin/bugate-role recover docs/usecases/<UC> \
  --lineage-id <exact-lineage-id> --expected-head <exact-head-or-EMPTY> \
  [--archive <trusted-recovery-archive>]
```

`lineage-init` 是 per-UC 治理动作，与 fresh-install-only 的
`scripts/bugate_init.py` 无关。`lineage-adopt` 只接受 exact head 上非空、完整通过
验证的 legacy chain，并报告 `receipts_rewritten: 0`。

Initialization 不会在首个 registry row 前留下无 journal 的缺口。它在第一次 Memory
request 前持久化唯一 exact intent，再按
`pending` -> `root_absence_verified` -> `root_verified` ->
`registry_initialized` -> `chain_written` -> `completed` 推进。Required mode 证明 root
不存在并创建、exact-verify deterministic root；best-effort 走同一 journal，但使用明确的
no-remote-root 边界。两种 mode 都在 completed 前以 `0600` 发布并 exact-verify empty
chain。Intent 创建后的 crash 报告 `recovery_pending`；重跑相同 exact `lineage-init`
会从原 stage 继续。初始 probe 若发现 strict root，则 abort 尚未创建 root/lineage 的
intent 并报告 `migration_required`。普通 lifecycle publisher 不能绕过 active
initialization。

Recovery 在写入前校验完整 source、每个现有 target，以及将恢复的 exact
receipt/chain bytes 和 mode。`active_initialization` 必须通过 exact `lineage-init`
继续，而不是运行 `recover`。Validation 与 write preflight 通过后，`recover` 会选定
active source 或创建 pending `recovery_restore`，claim 这笔 exact source，然后才恢复
exact committed predecessor、receipt/chain bytes 与 mode；live claimant 保持独占，
dead owner 只有经过 liveness check 与 exact-token CAS 才可 takeover。若原 lifecycle publication 已经具备
exact receipt 与经过验证、可进入 CAS 的 checkpoint，recovery 会让同一 transaction
完成 CAS 与 local publication，不会制造重复的 next-sequence lifecycle receipt。随后
registry 在同一个 SQLite transaction 中 terminalize 已 claim source：
`recovery_restore` 变为 `aborted`，lifecycle 变为 `completed`；同时安装唯一
canonical-bound、pending 的 `evidence_recovery` successor，因此没有
`aligned`/no-audit crash gap；若 active transaction 已是 `evidence_recovery`，retry 会
直接继续它，不会再安装 successor。恢复过程可见，不是静默 reset。Best-effort
journal 中的 durable empty transition ID 会
作为 unanchored marker 保留，不重试 Memory HTTP。

## 事务化 publication

每个 lifecycle event 现在都按持久顺序执行：

1. 获取 physical per-UC transition lock，并要求 integrity 为 `aligned`；
2. 按 exact head/sequence/revision/checkpoint 创建唯一 registry transaction，并在任何
   Memory request 前绑定 canonical transition；
3. required 模式下，对照 registry retained payload 与 exact head/sequence/state/revision，
   exact-verify deterministic root 和当前 committed predecessor checkpoint；
4. 按 `memory_mode` 准备 transition Memory record；
5. 构造 receipt，用 receipt hash finalize Memory；required 模式还要 exact-verify
   最终 transition/receipt binding；
6. 然后才在 registry 中单次冻结并绑定 exact receipt bytes/path/mode/hash，并构造
   exact minimal-chain bytes；
7. required 模式下，发布并 exact-verify 一份携带 exact receipt/chain envelope 的
   immutable checkpoint；
8. 对 registry head 执行 compare-and-swap；
9. 以 no-replace 语义发布 receipt，并原子替换 `chain.json`；
10. 只有本地 chain 已存在后，才把 transaction 标为 completed。

Predecessor exact verification 位于 pending journal 之后、new transition prepare 之前。
Checkpoint publication 位于 strict Memory finalize 与 exact registry receipt-byte bind
之后，绝不能早于或替代其中任一步。

Durable lifecycle-publication journal 创建后的任一中断都会表现为
`recovery_pending`；在此之前的 validation 或 handoff verification 失败不会创建新的
pending transaction。Initialization 使用上文独立的 intent-first journal。Sole active
transaction 在 Memory 操作前串行化竞争者，registry CAS 则保证同一确定性
lineage 的第二份 workspace copy 不能同时赢得同一个 head。

## Required 与 best-effort Memory

- `memory_mode: required` 会创建并 exact-verify 确定性 Memory root 和 immutable
  checkpoint chain。只要 registry 与 Memory history 仍在，就能从 checkpoint 重建
  缺失的本地 evidence。显式提供 trusted archive 只用于选择候选 bytes；retained
  strict checkpoint 仍是必须存在的权威，每个 archive envelope 都必须在任何写入前
  与其精确一致。它不是 strict Memory 不可用或分歧时的离线回退。
- `memory_mode: best_effort` 仍使用 machine registry 做删除检测和 publication CAS，
  也可能尝试 transition Memory call 并容忍失败，但不创建 remote lineage
  root/checkpoint。因此本地历史丢失后的 recovery 必须使用 `--archive` 提供一份单独
  保留、可信的 `bugate.role-recovery-archive/v1`。

Lineage 注册后修改配置中的 Memory mode 属于 integrity drift，不能被当作静默的
durability upgrade/downgrade。

## Updater 与 migration 边界

Imported updater 可以安装具备 v0.4.3 能力的 engine，但绝不编辑或创建 profile、
namespace、`00_role_evidence/**`、machine lineage registry 或 Memory data，也绝不运行
`lineage-init`、`lineage-adopt` 或 `recover`。

因此，updater `apply`/`verify` 成功只代表 managed engine projection 已安装，
**不代表**已有 UC 已接受 lineage migration。Engine update 经审查并提交后，要对每个
受治理 UC 运行 `lineage-status`，再分别执行其证据所要求、独立审查的 exact
init/adopt/recovery 路径。

v0.4.2 tag、Release 与 assets 保持不可变。v0.4.3 作为新的 patch release 交付；updater
只运输新 engine，不重写上一版 release，也不静默迁移 SUT-owned state。

## 安全与限制

Hook 仍无法拦截任意 shell redirection、递归删除或 external editor。Registry 加 strict
Memory 能检测 deletion/drift，但不能认证 operator，也不提供不可抵赖身份。更强控制
需要独立 OS 账号、容器、managed runner、受保护备份或 role-scoped server
credential。

拥有同一 OS-user 权限的 actor 如果同时删除 workspace evidence、machine registry 和
完整 Memory home，就移除了全部本地锚点；这种组合破坏超出 BUGate 的本地威胁边界。
同样，在确定性 registry/root 建立前就已遗失的历史，无法从空目录反向推断。应恢复
可信的 pre-loss evidence，或保持 migration 阻断；绝不能制造缺失的过去。

## 发布门禁

Durable defect 记录已经保存 source-level lineage/Memory/hook/updater 回归、complete
direct discovery、de-SUT checks、双语文档契约与独立 synthetic probes。在这些源码
bytes 最终化时，独立语义评审、clean archive-native `smoke + both`、merged-main CI、
annotated-tag CI 与 public-download reacceptance 仍是待完成的 release operation gate；
任一 gate 失败都必须停止发布。

正式 archive-native gate 必须使用精确参数
`--full-check-mode smoke --full-check-archive both`，让 tar 与 zip 都执行 lineage
flow、updater preservation、rollback/reapply 与 ownership oracle。该 smoke route
会故意中断 strict Memory publication，再通过 `recover` 继续；因此 acceptance
必须绑定按顺序的六个 lifecycle event 与一条保持状态的 `evidence_recovery`
receipt，共七个 exact anchor，lineage sequence/revision 为 `0..6`。即使
archive runtime 本身已经通过，仍按六次计数的旧 release harness 也必须
`NO-GO`。对应 release-infrastructure defect 与 regression 记录见
[`BUGATE-CORE-2026-07-23-RELEASE-ACCEPTANCE-TRANSITION-COUNT`](../defects/BUGATE-CORE-2026-07-23-RELEASE-ACCEPTANCE-TRANSITION-COUNT.md)。
语义 release
review 保持 provider-neutral，且至少需要两个真实、独立 session。优先跨 provider；
一个 provider 不可用时，只接受**新建**的**同源** session 作为 fallback，并要求它
使用不同 prompt，且 synthesis 前不能看到第一个结论。Placeholder、延续同一会话、
self-review 或把 optional provider outage 改写成成功，都不能满足该 gate。

Release distribution 必须且只能包含：

```text
bugate-0.4.3.tar.gz
bugate-0.4.3.zip
bugate-0.4.3.SHA256SUMS
```

最终 SHA-256 只能从干净的 final commit 构建后生成，并写入
`bugate-0.4.3.SHA256SUMS` 与已发布 GitHub Release record。本文不会嵌入自身 archive
hash，也不声称 assets 已经存在。
