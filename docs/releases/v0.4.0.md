# BUGate v0.4.0

[English](v0.4.0.md) | [简体中文](v0.4.0.zh-CN.md)

Release date: 2026-07-20

BUGate v0.4.0 adds opt-in, auditable Wave 7 lifecycle-role governance while
preserving v0.3.x behavior for profiles that do not enable it. The normative
contract is [the Wave 7 role-governance protocol](../qa-methodology/ROLE_GOVERNANCE_PROTOCOL.md).

## Highlights

- A fail-closed `designer` → `implementer` → `reviewer` state machine now
  governs pre-code, implementation, and post-run ownership. `bin/bugate-role`
  provides `status`, `verify`, `approve`, `handoff`, `accept`, `complete`,
  `session-start`, and isolated `run` commands.
- Every transition appends a hash-linked local receipt under
  `<artifact-dir>/00_role_evidence/`. Receipts snapshot the active profile and
  relevant artifacts; implementation and execution evidence are added at the
  corresponding boundaries. Profile, artifact, implementation, receipt, or
  chain drift is detected before a governed write is admitted.
- `memory_mode: required` makes role transitions use strict Memory writes and
  exact-ID reads. Any unavailable service, timeout, write failure, missing ID,
  namespace/role/UC/phase/hash mismatch, or failed receipt binding returns
  non-zero and publishes no unlocking local receipt or chain head. Ordinary
  recall, notes, and Stop heartbeats still contact Memory but remain
  best-effort; per-edit role checks are local and make no network request.
- The orchestrator and Core artifact mutators run the same role preflight as
  the hooks. `--auto` reports lifecycle states such as
  `READY_FOR_HUMAN_ACCEPTANCE`, `READY_FOR_DESIGNER_HANDOFF`,
  `IMPLEMENTATION_UNLOCKED`, `READY_FOR_REVIEWER_HANDOFF`, `POST_RUN_ACTIVE`,
  and `CLOSED`; it never auto-accepts 03B.
- Claude and Codex hook surfaces include `check_role_evidence.py` alongside the
  existing pre-code guard. Direct agent-tool edits to `00_role_evidence/**`
  are denied. SessionStart prints the active mode/role/session; Stop remains a
  best-effort heartbeat.
- Wave 1 peer subprocesses now strip `BUGATE_AGENT_ROLE`,
  `BUGATE_SESSION_ID`, and receipt/session identity variables while retaining
  proxy, model, effort, profile, and project-root settings. A Wave 1 peer is an
  independent design reviewer, not a Wave 7 lifecycle actor.
- Config loading now preserves nested mappings with deterministic deep merge:
  mappings merge recursively, profile scalars replace base scalars, and
  profile lists replace base lists. Targeted v0.3.x aliases remain supported
  for `memory.namespace`/`namespace` and `bugate.mode/version`/top-level
  `mode/version`; generic nested keys are not flattened.

## State machine

| Event | Required boundary | Result |
|---|---|---|
| `human_acceptance` | A designer records an already-made human decision; configured 03B is already `gate_status: passed` | `ready_for_designer_handoff` |
| `designer_handoff` | Current pre-code/provenance snapshot plus strict Memory anchor | `awaiting_implementer_acceptance` |
| `implementer_acceptance` | A distinct implementer session accepts the exact handoff Memory ID | `implementation_unlocked` |
| `implementer_handoff` | At least one guarded, same-UC implementation file is hash-snapshotted | `awaiting_reviewer_acceptance` |
| `reviewer_acceptance` | A distinct reviewer session accepts the exact handoff and current implementation snapshot | `post_run_active` |
| `reviewer_completion` | 04/05, command/exit code, execution evidence hashes, and final gates are recorded | `closed` on a passed result |

Receipts use `bugate.role-evidence/v1`; `chain.json` uses
`bugate.role-chain/v1`. JSON is canonical UTF-8 with sorted keys and compact
separators, and receipt hashing excludes `receipt_sha256` itself. Publication
uses same-directory temporary files, `fsync`, and `os.replace`.

## Compatibility and migration

- Profiles without `role_governance`, or with `role_governance.mode: off`,
  behave as v0.3.x. Core still ships with `mode: off` and is not self-locked.
- `agent_roles` remains an independent path read/write policy. It can be used
  without lifecycle governance and is not merged into `role_governance`.
- Enabling `mode: required` makes an unset/wrong role or required session ID a
  hard block. Historical passed UCs are not grandfathered: record a current
  human acceptance, designer handoff, and implementer acceptance before Layer
  4; post-run likewise needs implementer handoff and reviewer acceptance.
- Re-run `bugate_init.py` to refresh vendored scripts and BUGate-owned hook
  entries while preserving SUT-owned hooks. Start designer, implementer, and
  reviewer as three fresh sessions. A hook subprocess cannot export variables
  into its parent, and an already-running Desktop app does not acquire shell
  environment changes; relaunch it from the intended environment.
- Do not run `--auto` again after a human has passed 03B. Record that existing
  decision with `bugate-role approve`, then hand off directly.
- A Memory outage blocks only the next strict role transition, not every edit.
  Restore service health and retry; retries are idempotent. Profile/pre-code
  drift requires a new human/designer generation. Implementation drift
  requires a new implementer handoff and reviewer acceptance. Never delete or
  hand-edit evidence to reset it; restore tampered receipts from trusted
  evidence before appending a superseding transition.

## Distribution and verification

The GitHub Release contains exactly three assets:

```text
bugate-0.4.0.tar.gz
bugate-0.4.0.zip
bugate-0.4.0.SHA256SUMS
```

Download the checksum file with the archive and verify SHA-256 before
extracting. The tar archive is preferred where preserving symlinks matters;
the zip contains the equivalent kit.

## Security and activation limits

`approved_by` is a declaration that records an existing human decision; it is
not authentication. Environment roles, local hooks, local files, hash linkage,
and external Memory anchors provide auditability and tamper/drift detection,
not non-repudiable identity. Strong identity isolation requires separate OS
accounts, containers, managed runners, or role-scoped server credentials.

Hooks cannot intercept arbitrary shell redirection or writes by an external
editor. Supported agent tools, the orchestrator, and Core artifact mutators are
enforced; stronger filesystem control is a managed-runner responsibility.
Finally, the Codex hook hash changes in this release. Codex Desktop hooks can
remain silently inactive until the imported project is explicitly re-trusted;
file/re-vendor verification alone is not proof that runtime enforcement is
active.
